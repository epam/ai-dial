# Collaboration and sharing

DIAL is a collaboration platform: users create resources such as applications, prompts, conversations, and tool sets, then make them available to colleagues or the whole organization under controlled access. This page explains the model behind collaboration in DIAL — private and public spaces, sharing, and publication — and how those mechanisms relate. It is written for developers and evaluators who want to understand the model, not yet to configure it.

## Collaboration rests on access control

Everything DIAL Core manages is a [resource](../2.architecture/1.architecture-highlights.md) — an application, conversation, prompt, file, or tool set. Each resource starts in its owner's **private space**, visible only to them. Collaboration is the controlled relaxation of that boundary: granting other users access to a resource without losing ownership of it.

Two distinct mechanisms relax the boundary, and the difference between them matters:

- **Sharing** grants access to specific users, one resource at a time, with no approval needed.
- **Publication** moves a resource into the **public space** for a broad audience, and always requires administrator approval.

Both build on DIAL's [access control](../../4.operating-dial/5.auth-and-access-control/0.index.md), so a collaborator only ever sees what the owner and the organization's roles permit.

## Sharing: targeted, owner-controlled access

Sharing lets a resource owner create a link and give it to chosen recipients. The resource stays in the owner's private space; the recipient gains access to that specific resource. The owner keeps control and can revoke access at any time.

Sharing is the right tool for working closely with a few people. You might share a conversation with a colleague to get feedback, or share an application with a teammate to test it. A sharing link can optionally grant **edit and re-share** permissions, which turns it into a real-time collaboration tool — for example, two developers iterating on the same [Code App](../../3.building-with-dial/1.apps/3.code-apps/0.index.md). To keep collaborators in sync, DIAL Core can notify subscribers when a shared resource changes.

Once shared, a resource becomes available to the recipient in DIAL Chat, in the [Marketplace](../../3.building-with-dial/5.developer-tools/2.chat-customization/6.marketplace.md), and through the API.

## Publication: organization-wide availability

Publication is how a resource graduates from a private project to a shared organizational asset. A published resource is placed in the public space, where it is available to all authenticated users by default; access rules on subfolders can restrict it to specific roles or groups. Published applications and tool sets appear in the [Marketplace](../../3.building-with-dial/5.developer-tools/2.chat-customization/6.marketplace.md), making it the collaboration hub for the organization.

Because publication exposes a resource widely, every publication request requires administrator approval. This review step is what makes the public space trustworthy: users can rely on published resources having passed a gate. Publication also enables versioning in practice — you can publish a stable version while continuing to develop the next one privately, without disturbing what others are using.

## Sharing and publication compared

| | Sharing | Publication |
|---|---|---|
| Audience | Specific users you choose | All authenticated users, or a restricted set by role |
| Approval | None | Administrator approval required |
| Where the resource lives | Stays in the owner's private space | Moved to the public space |
| Typical use | Feedback, co-development, small teams | Releasing a finished asset to the organization |

## Collaboration between deployments

Collaboration is not limited to people. Applications and tool sets — collectively, deployments — can also be granted access to resources or to each other, so an [agent](./1.agentic-platform.md) can act on a user's resources or call another deployment as part of a workflow. This is the same access-control model applied to non-human actors, and it is what lets multi-agent applications operate on shared data safely.
