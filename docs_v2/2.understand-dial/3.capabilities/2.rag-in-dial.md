# RAG in DIAL

Retrieval-augmented generation (RAG) grounds model answers in your own documents by retrieving relevant passages before the model responds. This page explains how DIAL approaches RAG, which components are involved, how its retrieval differs from a naive vector search, and when to use DIAL RAG instead of building a pipeline yourself. It is written for developers who understand RAG in principle and want to know what DIAL provides.

## Why RAG matters for enterprise deployments

A language model only knows what was in its training data. RAG closes that gap by retrieving passages from a trusted corpus — policy documents, product manuals, code — and supplying them as context, so answers are grounded in current, organization-specific information rather than the model's parametric memory.

In an enterprise setting, the retrieval step is also a governance boundary. What a user can retrieve must respect the same access control as the rest of the platform, indexed content must live in storage the organization controls, and the whole flow needs the same observability as any other AI traffic. Bolting these properties onto a hand-rolled RAG pipeline is where most of the effort goes. DIAL provides them as platform features.

## DIAL's approach

[DIAL RAG](https://github.com/epam/ai-dial-rag) is EPAM's reference RAG implementation, shipped as a DIAL application. Because it is an application, it speaks the [Unified API](../2.architecture/4.unified-api-overview.md) like any other [agent](./1.agentic-platform.md): you attach documents to a chat request, and it returns a grounded answer. It reaches models and embeddings through [DIAL Core](../2.architecture/1.architecture-highlights.md) rather than calling providers directly, so it inherits the platform's model gateway, access control, and file storage.

The pipeline has three stages:

```text
Documents ──▶ Indexing ──▶ Retrieval ──▶ Generation ──▶ Grounded answer
            (parse +       (hybrid       (chat model
             build index)   search)       + context)
```

- **Indexing** parses each attached document and builds a searchable index. DIAL RAG accepts PDF, DOC/DOCX, PPT/PPTX, TXT, and other plain-text formats including code files, plus image formats such as PDF, JPEG, and PNG for visual understanding.
- **Retrieval** finds the passages most relevant to the user's question.
- **Generation** passes those passages to a configurable chat model (GPT-4.1 by default; AWS and GCP deployment profiles use Claude Sonnet and Gemini instead) to compose the answer.

### Hybrid, multimodal retrieval

DIAL RAG does not rely on vector similarity alone. It combines up to four retrievers and merges their results, which improves recall across different document and query types:

- **Semantic retriever** — text-embedding similarity, using a locally hosted `bge-small-en` model. Good for paraphrased and conceptual queries.
- **Keyword retriever** — Okapi BM25 lexical ranking. Good for exact terms, identifiers, and rare words that embeddings can miss.
- **Description retriever** — a vision model describes each page image in text, and those descriptions are searched. This surfaces information that lives in diagrams, tables, and scanned pages.
- **Multimodal retriever** — multimodal embeddings index the page images directly for visual similarity search.

The image-based retrievers can be turned off for a text-only mode when the answer model does not accept images. Combining lexical, semantic, and visual signals is DIAL RAG's main advantage over a pipeline that indexes embeddings only.

### Index storage and reuse

Indexing is the expensive step, so DIAL RAG is built to do it once. Indexes can be stored in DIAL [file storage](../../3.building-with-dial/7.working-with-dial-resources/1.file-management.md) and reused across requests, with an in-memory cache in front to avoid repeated fetches. Indexes are versioned and migrated automatically as the format evolves, and DIAL RAG checks each document's ETag and modification time to decide when a re-index is actually needed. Repeat questions against the same corpus skip parsing entirely.

### Using parts of the pipeline

DIAL RAG exposes three request modes, so you can adopt as much of the pipeline as you need:

- **Full RAG** — index (if needed), retrieve, and generate the answer. The default.
- **Indexing only** — produce reusable index files and manage them yourself.
- **Retrieval only** — return the relevant passages without generating an answer, so you can run your own generation step on top of DIAL's retrieval.

The retrieval-only mode is the middle ground between adopting DIAL RAG wholesale and building everything yourself: DIAL handles parsing, indexing, and hybrid search, while your application keeps control of the final prompt and model.

## DIAL RAG and DIAL RAG Eval

[DIAL RAG Eval](../../3.building-with-dial/4.evaluations/1.rag-eval-toolkit.md) is a separate, provider-agnostic library — not a runtime part of DIAL RAG. Where DIAL RAG answers questions, RAG Eval measures how well a RAG application does so: retrieval metrics such as recall, precision, and mean reciprocal rank, and generation metrics such as faithfulness and answer coverage scored by an LLM judge. You use it offline, during development, to tune retrieval and prompting — the evaluation half of an eval-driven build loop, applicable to any RAG application, not only DIAL RAG.

## How DIAL RAG differs from LangChain or LlamaIndex

Frameworks such as LangChain and LlamaIndex give you primitives to assemble a RAG pipeline from scratch. DIAL RAG gives you an assembled, governed pipeline. The trade-off is control versus integration:

| DIAL RAG provides | Raw frameworks provide |
|---|---|
| A working multimodal RAG application integrated with DIAL access control, file storage, and the model gateway | Full control over chunking, custom or graph retrieval, and bespoke pipeline shapes |
| Index caching, versioning, and migration handled for you | Freedom to deploy outside DIAL on any stack |
| Distribution as a governed agent in the [Marketplace](../../3.building-with-dial/5.developer-tools/2.chat-customization/6.marketplace.md) | Direct use of the latest framework features as they ship |

A useful rule of thumb: reach for DIAL RAG when you want grounded answers over your documents with platform governance and little infrastructure to run. Build your own — or use DIAL RAG's retrieval-only mode as a component — when your pipeline needs structure or retrieval strategies DIAL RAG does not express.

## Trade-offs and limitations

- DIAL RAG's retrieval strategies are configurable but fixed in kind. If you need a fundamentally different approach — graph-based retrieval, agentic multi-hop retrieval, or a custom reranking model — a custom pipeline gives more room.
- It is designed to run inside DIAL. Using it standalone discards the integration that is most of its value.
- Indexing large documents takes time and compute on first encounter; the cost is amortized by index reuse, but the first query against a new corpus is slower.
