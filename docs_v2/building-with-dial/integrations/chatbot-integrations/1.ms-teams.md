# Integration with Microsoft Teams

This page explains how DIAL integrates with Microsoft Teams through a custom bot, so users can reach DIAL's models and applications without leaving their primary communication tool. It is written for developers who want to understand the architecture and design decisions before building a similar integration. It describes the approach at a conceptual level rather than providing step-by-step deployment instructions.

**Note**
> A hands-on, step-by-step tutorial — with a complete sample repository, file-by-file walkthrough, and Azure setup steps — is planned. It will be published once a paired sample bot is available. This page covers the architecture and design rationale in the meantime.

## The problem and the design goal

Modern workplaces rely on many specialized tools, and constantly switching between applications disrupts workflow. Users need access to DIAL's capabilities without leaving the environment where they already communicate. A custom Microsoft Teams bot creates a bridge between the two platforms.

We designed the integration around three goals:

- Provide a consistent user experience across environments.
- Work in both private and group chat contexts.
- Handle authentication appropriately for each context.

The result is that Microsoft Teams users can use DIAL's models and tools directly inside their conversations, while DIAL continues to enforce its existing role-based access control.

## What the bot does

The bot brings selected DIAL functionality into Microsoft Teams chats while respecting each user's permission level:

- Use any model or application available in DIAL, subject to the user's permissions.
- Access existing DIAL conversations directly in Microsoft Teams.
- Manage conversations — clear history, switch topics — and use the [Playback](../../../chat-user-guide/1.conversations.md) feature.

## Architecture

The integration has three main components:

1. **Microsoft Teams client** — the Teams UI the user interacts with.
2. **DIAL Chat Bot Service** — a middle layer built with the [Azure Bot SDK](https://learn.microsoft.com/en-us/azure/bot-service/index-bf-sdk?view=azure-bot-service-4.0). It proxies traffic between Teams and DIAL.
3. **DIAL Core** — the backend that exposes the [Unified API](https://dialx.ai/dial_api) and performs all model orchestration.

These components are connected through two Microsoft services:

- [Azure Bot Service](https://azure.microsoft.com/en-us/products/ai-services/ai-bot-service) handles the messaging infrastructure and initial authentication.
- The [Microsoft Teams app manifest](https://learn.microsoft.com/en-us/microsoftteams/platform/resources/schema/manifest-schema) defines the bot's presentation and capabilities within Teams.

The request path is:

```text
[Microsoft Teams] <--> [Azure Bot Service] <--> [DIAL Chat Bot Service] <--> [DIAL Core]
```

### Conversation handling

The DIAL Chat Bot Service acts as a proxy that converts communication between the Teams client and DIAL. The bot receives events from Teams through the Azure Bot Service, which handles authentication and token verification first. The Azure Bot Service then delivers clean events to the service — for example, a notification that a specific user sent a message in a particular chat.

The service forwards these requests to DIAL Core through the Unified API, then transforms DIAL's responses into a Teams-compatible format. Because Teams does not support direct HTML formatting, the service uses [Adaptive Cards](https://learn.microsoft.com/en-us/microsoftteams/platform/task-modules-and-cards/cards/design-effective-cards?tabs=design) to render interactive elements such as combo boxes and checkboxes.

The bot maintains conversation context by mapping Teams conversation IDs to the corresponding DIAL sessions, which preserves continuity across interactions.

### Media file handling

Media handling addresses the different ways Teams transfers files. When users send images in Teams, small images may be embedded directly in the message, while larger files are uploaded to Microsoft SharePoint with a link sent to the bot. The bot detects each format, retrieves the file, and uploads it to DIAL.

For files that flow from DIAL back to Teams, the service uses [Azure Blob Storage](https://azure.microsoft.com/en-us/products/storage/blobs) as an intermediary. Files are uploaded to Blob Storage, and Shared Access Signature (SAS) tokens generate secure, time-limited URLs that Teams can display.

SAS tokens expire. When a user returns to a conversation after the token has expired — typically after a period such as a month — the files are no longer visible in the Teams chat history, although they [remain available](https://dialx.ai/dial_api#tag/Files) in DIAL itself.

## Authentication and authorization

The bot uses two authentication approaches, chosen by conversation context. For background on how DIAL handles authentication and authorization, see [Auth and access control](../../../operating-dial/auth-and-access-control/0.index.md).

- **Private chats** — the bot acts on behalf of the individual user. The Azure Bot Framework manages the login flow with the configured identity provider, obtains a token, and the bot uses that token to authenticate with DIAL. For the identity providers DIAL supports, see [SSO / IdP setup](../../../operating-dial/auth-and-access-control/sso-idp/0.index.md).
- **Group chats** — the bot uses a pre-configured API key instead of individual credentials. This gives every group member a consistent experience but limits access to the models and applications the API key is permitted to use.

The Azure Bot Service handles token management and authentication flows, which lets the bot implementation focus on relaying information between Teams and DIAL.

## Deployment components

A production deployment of the bot relies on the following infrastructure:

- [Azure Bot Registration](https://learn.microsoft.com/en-us/azure/bot-service/bot-service-quickstart-registration?view=azure-bot-service-4.0&tabs=userassigned) — configured with the messaging endpoint for API communication.
- [Azure App Service](https://learn.microsoft.com/en-us/azure/app-service/) — hosts the bot and holds its App ID and client secret.
- OAuth connections — a DIAL connection for user authentication.
- [Microsoft Graph](https://developer.microsoft.com/en-us/graph) — used to access files stored in SharePoint.
- [Azure Key Vault](https://learn.microsoft.com/en-us/azure/key-vault/general/overview) — stores DIAL API keys, Azure App credentials, and storage connection strings.
- Kubernetes — runs the containerized bot service with environment-specific configuration and ingress for HTTP traffic with TLS.

## Technical stack

The reference implementation uses:

- **Bot framework** — Azure Bot Service / [Azure AI Foundry](https://azure.microsoft.com/en-us/products/ai-foundry).
- **SDK** — [Azure Bot SDK](https://learn.microsoft.com/en-us/azure/bot-service/index-bf-sdk?view=azure-bot-service-4.0) (.NET in this implementation).
- **Authentication** — [Microsoft Identity Platform](https://learn.microsoft.com/en-us/entra/identity-platform/) and [OAuth 2.0](https://oauth.net/2/).
- **File storage** — [Azure Blob Storage](https://azure.microsoft.com/en-us/products/storage/blobs).
- **Communication** — REST APIs against the DIAL Unified API.
