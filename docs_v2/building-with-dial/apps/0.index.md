# DIAL Apps overview

DIAL is an application server for GenAI applications. Apps are first-class citizens on the platform — they are composable, discoverable, and governed by the same rules as language models. This page explains what DIAL apps are, how they relate to each other, and how the platform manages them.

## What makes an app "DIAL-native"

A DIAL-native application is any service that speaks the [Unified API](../../understand-dial/architecture/4.unified-api-overview.md) — an OpenAI-compatible protocol that DIAL Core uses for all communication with models and applications. Because every app speaks the same protocol:

- Any app can call any other app as an agent in a multi-agent workflow.
- Models and apps are interchangeable from a caller's perspective.
- Platform-level governance — authentication, rate limits, logging, load balancing — applies uniformly to all apps and models without extra integration work.

The Unified API supports tool calling, streaming, multi-modality, seeds, and token usage reporting. Apps built with the [DIAL SDK](https://github.com/epam/ai-dial-sdk) get protocol compliance out of the box.

## Two categories of apps

DIAL distinguishes between **Schema-based apps** and **Custom Apps** (apps without schemas).

### Schema-based apps

Schema-based apps are defined by a JSON schema that describes their structure, properties, and endpoints. The schema enables:

- **UI wizards** — end users can create and configure app instances through DIAL Chat without writing code.
- **API-driven configuration** — developers and admins can create or modify instances through the DIAL Core API.
- **Platform-managed hosting** — a single runner container serves all instances of a given app type. The platform handles deployment, scaling, and lifecycle.

DIAL includes three standard application types:

| Application type | What it does |
|---|---|
| [**Quick App**](quick-apps/0.index.md) | No-code orchestrator that combines a model, instructions, and tools into a workflow |
| [**Code App**](code-apps/0.index.md) | Python app written and deployed directly in the DIAL Chat browser editor |
| [**Mind Map Studio**](mind-map-studio/0.index.md) | Interactive knowledge graph that explores documents and URLs |

Organizations can also define custom application types with their own schemas, UIs, and editors.

### Custom Apps

Custom Apps are not tied to any application type schema. You write the application code, containerize it, deploy it alongside DIAL Core, and register it by providing the endpoint URL. Business logic lives in the container — it cannot be modified through the DIAL Core API after deployment.

Custom Apps are the right choice when you need full control over dependencies, network access, infrastructure, and runtime behavior.

See [Custom Apps: architecture and lifecycle](custom-apps/0.index.md) for details.

## The app lifecycle

Every DIAL app follows the same high-level lifecycle, regardless of type:

1. **Develop** — Write code with the DIAL SDK, configure a schema-based app through the UI, or write Python in the Code App editor.
2. **Register** — Schema-based apps are registered via the DIAL Core API or UI wizard. Custom Apps are registered in `config.json` with an endpoint URL.
3. **Test** — Use the app through DIAL Chat, the Unified API, or the evaluation toolkit.
4. **Share or publish** — Share with specific users (READ or WRITE access) or publish to the Marketplace for broader access. Publishing requires administrator approval.
5. **Operate** — Monitor usage through Analytics Realtime and observability tools. Manage access, update configuration, and scale as needed.

## The Marketplace

The [Marketplace](../developer-tools/chat-customization/6.marketplace.md) is where users discover and activate apps. Published apps appear in the Marketplace alongside language models. Users can browse, search, and add apps to their workspace.

The publication workflow ensures quality: a developer submits a publication request, an administrator reviews and approves it, and the app becomes available to the designated audience (specific roles or all users).

## Middleware and platform services

DIAL apps benefit from a shared middleware layer without additional integration:

- **Language models** — connect to OpenAI, Bedrock, Vertex AI, self-hosted models, and more through [adapters](../../understand-dial/architecture/1.architecture-highlights.md#ai-model-adapters).
- **Interceptors** — add custom logic to requests and responses (PII redaction, guardrails, safety checks).
- **Access control** — RBAC integrated with identity providers.
- **Rate limits** — per-key and per-role usage and cost controls.
- **Load balancing** — distribute requests across upstream resources with PTU support.
- **Observability** — OpenTelemetry-powered metrics, tracing, and logging.
- **Evaluation** — RAG evaluation toolkit for retrieval and generation quality metrics.
