# Usage limits and cost control

DIAL meters how much each caller consumes and stops requests that exceed their allowance. This page explains the two limit types — token rate limits and cost limits — how DIAL enforces them, and how usage is monitored. It is written for architects and operators who want to understand the governance model before configuring it. Limits attach to [roles](./1.authentication-and-access-control.md), so a reader should first understand authentication and access control.

## Why limits exist

An open gateway to language models is also an open path to runaway spend and resource contention. Limits let an organization manage consumption, prevent overuse, and allocate model capacity fairly across users and applications. DIAL applies limits at the level of a single user, a group of users, or an API key, always through the roles those subjects hold.

DIAL offers two independent controls. Token rate limits govern volume; cost limits govern money. They address different concerns, and most deployments use both.

## Token rate limiting

Token rate limiting caps the volume of tokens a subject may process within a time window. DIAL counts both input and output tokens and enforces limits across minute, daily, weekly, and monthly intervals. When a subject reaches a limit, further requests are rejected until the window resets.

The purpose is fair resource distribution and tiered service. Administrators set different token allowances for different roles — a generous allowance for a production application, a smaller one for exploratory use.

## Cost-based rate limiting

Cost-based rate limiting sets a monetary ceiling on consumption. Token limits have uneven financial impact because models differ in price; a cost limit instead caps actual spend across every model a role can reach.

DIAL calculates the cost of each interaction as it happens, tracking currency-based consumption across the configured interval. Because the calculation accounts for each model's pricing automatically, an organization gets a consistent budget control regardless of which models its users choose.

## How limits are enforced

DIAL Core checks every request against both controls before it reaches a model. The two operate in parallel, and a request must satisfy all applicable limits to proceed. When a request is rejected, the caller receives a message identifying the specific limit that was triggered.

This dual check keeps both technical capacity and financial budgets within bounds during model use. DIAL Core records each limit check, so the decisions are available for auditing and analysis.

## Monitoring and analytics

Limits are only as good as the visibility into them. The [Analytics Realtime](https://github.com/epam/ai-dial-analytics-realtime) component tracks consumption patterns and limit enforcement over time. The data flows into time-series databases and is visualized through tools such as Grafana and Power BI.

This visibility closes the loop: administrators see actual usage against configured limits and adjust allowances to match real demand rather than guesswork.
