# Access control reference

This page documents the authorization rules DIAL Core enforces for every object type and subject. It lists access patterns, the access-control lists (ACLs) for objects, special objects, and Core APIs. For the concepts behind these tables — objects, subjects, actions, and spaces — see [Authentication and access control](./1.authentication-and-access-control.md).

## Authorization rule notation

DIAL authorization is expressed as a rule granting a set of subjects a set of actions on a set of objects:

```text
(object: {set-of-objects}, subject: {set-of-subjects}, action: {actions})
```

This notation is a documentation convention, not the internal storage format. The tokens below appear throughout the rules.

| Token | Meaning |
|---|---|
| `public:*` | Any object in the public space. |
| `private:subject_id:*` | Any object in the private space of the subject identified by `subject_id`. |
| `private:subject_id:app_id:*` | Any object in the application `app_id` subfolder of a subject's private space. |
| `private:app_id:*` | Any object in the application's own private space. |
| `subject: *` | Any subject. |
| `role:admin` | Any subject in the DIAL Administrators role. |
| `subject_id` | A unique identifier of a single user or API key. |

### Default rules

| Rule | Effect |
|---|---|
| `(object: public:*, subject: *, action: read)` | Read-only access to public objects for everyone. |
| `(object: public:*, subject: role:admin, action: *)` | Full access to public objects for DIAL Administrators. |
| `(object: private:subject_id:*, subject: subject_id, action: *)` | Full access to a subject's own private objects. |

## Access patterns

| Space | Access pattern | Description |
|---|---|---|
| Public space | Public objects | Read-only access to public objects without restrictions. For executable objects such as applications, routes, or tool sets, this also includes execute and call permissions. |
| Public space | Restricted objects | Read-only access to objects with matching rules. |
| Public space | Admin access | Full read/write access to any object in the public space; management of DIAL Core configuration files; approval and rejection of publication requests. |
| Private space | Own objects | Full read/write access to any object in the subject's own private space; submission of publication requests; creation of sharing links. |
| Private space | Own objects per application | Full access to any object in an application-specific subfolder within the subject's own private space. |
| Private space | Application objects | Full access to any object in the application's own private space. |
| Private space | Shared objects | Access to a shared object at the level defined in the sharing link: read, read/write, and optionally re-share. |

## ACLs for special objects

Special objects are part of DIAL's authorization system and belong to neither the public nor the private space. A `+` denotes access; a `-` denotes no access.

| Object | DIAL Administrator | User without role | User / API key with role(s) | Per-request key |
|---|---|---|---|---|
| **Keys** — API keys are managed only via DIAL Core configuration files. Keys cannot be listed or accessed via the API, even read-only. A caller authenticated by an API key can call [`/v1/user/info`](https://dialx.ai/dial_api#operation/getUserInfo) to obtain limited information about the key. | + | - | - | - |
| **Roles** — managed only via configuration files. Roles cannot be listed or accessed via the Core API, even read-only. | + | - | - | - |
