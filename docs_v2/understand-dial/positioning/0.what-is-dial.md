# What is DIAL

DIAL is an open-source, enterprise-grade platform that combines a vendor-agnostic gateway to language models, an application server for building and running AI apps, and the access control, collaboration, and observability features needed to operate them in production. This page explains what DIAL is, the problem it solves, and where it fits next to the other tools in an AI stack. It assumes no prior DIAL knowledge.

## The problem DIAL solves

Enterprise GenAI teams rarely struggle with a single model call. They struggle with everything around it: consuming models from several providers and clouds without rewriting clients, hosting the applications built on top of those models, storing conversations and files, controlling who can access what, sharing and publishing work across an organization, and observing the whole system in production.

Most tools address one slice of this. A gateway normalizes provider APIs. A framework helps you compose logic in code. A studio gives one vendor a console. DIAL provides the platform that ties these concerns together, and it is API-first, so each capability is also an extension point.

## What DIAL provides

DIAL is built around **DIAL Core**, the only mandatory component. Core exposes the platform's capabilities through one consistent interface; every other component is optional.

| Capability | What it gives you |
|---|---|
| **Unified API** | One OpenAI-compatible protocol to reach every language model, embedding model, and application — regardless of provider. |
| **Application server** | A runtime that hosts first-class applications: Custom Apps (full code), Quick Apps (no-code orchestration), and Code Apps. |
| **Adapters** | Translation of external provider APIs (Azure OpenAI, AWS Bedrock, Google Vertex AI, and OpenAI-compatible endpoints) into the Unified API. |
| **Interceptors** | In-flight middleware over requests and responses for PII redaction, filtering, and compliance. |
| **Resource storage and access control** | Resilient storage for files, conversations, prompts, and applications, with role-based access control. |
| **Collaboration** | User-to-user sharing, organization-wide publications, and administrator review workflows. |
| **DIAL Chat** | A production web UI for end users: conversations, the Marketplace, applications, and files. |
| **Governance** | Per-role rate limits, cost control, usage limits, and authentication via JWT, API keys, and SSO/IdP. |
| **Observability** | OpenTelemetry-based tracing, metrics, and logs across Core, adapters, interceptors, and applications. |

The Unified API is the unification layer that makes all models and applications interchangeable. The application server is what turns that gateway into a place to build and run software, not merely route requests through it.

## Where DIAL fits

DIAL is designed to **enhance and extend**, not to replace or compete with, the rest of an AI stack. It complements:

- **Model providers and vendors** — Azure, AWS, GCP, OpenAI, Anthropic, Google, Meta, Mistral. DIAL reaches their models through adapters.
- **AI studios** — Azure AI Foundry, Google AI Studio. DIAL can front a studio's models as one more provider while staying multi-cloud.
- **Application frameworks** — LangChain, LlamaIndex, Semantic Kernel. These run *inside* a DIAL application.
- **Low-code builders** — Dify, Flowise, n8n. DIAL's Quick Apps cover similar ground, and DIAL can sit as a node in an automation pipeline.

What distinguishes DIAL is the combination. Most tools do one of four things well — act as a gateway, build apps, govern access, or provide an end-user UI. DIAL provides all four as an integrated, open-source, self-hostable platform. The [comparisons](./1.comparisons.md) explain each relationship in detail.

## Design principles

DIAL's shape follows from a few deliberate choices:

- **Open-source.** Licensed under the [Apache License, Version 2.0](https://github.com/epam/ai-dial/blob/main/LICENSE).
- **Vendor- and cloud-agnostic.** Runs on any cloud or on-prem; no lock-in to one model vendor or cloud.
- **API-first.** Every feature is available through an API, so custom extensions build on the same surface as the platform itself.
- **Domain-agnostic.** DIAL prioritizes generic, reusable capabilities over features specific to a single industry. Domain logic belongs in applications and extensions.
- **Small footprint.** No centralized database — DIAL relies on file storage and Redis, and runs equally on a laptop or a cluster.

The name reflects this integrating role: DIAL stands for **D**eterministic **I**ntegrator of **A**pplications and **L**anguage Models.
