# DIAL vs frameworks

Application frameworks such as LangChain, LlamaIndex, and Semantic Kernel are in-process libraries for composing LLM calls, chains, agents, and retrieval in code. DIAL is a platform that hosts and operates the applications those libraries help you write. This page explains why the two are complementary rather than competing, and how they work together. It assumes familiarity with [What is DIAL](../0.what-is-dial.md).

## What frameworks do

A framework is an SDK that runs inside *your* application process. It gives developers abstractions for working with models:

- **LangChain** — chains, agents, tool calling, and memory across many models, with a large ecosystem.
- **LlamaIndex** — a data framework focused on RAG: ingestion, indexing, retrieval, and data connectors.
- **Semantic Kernel** — Microsoft's SDK for orchestrating models, plugins, and planners, with strong .NET integration.

These libraries answer "how do I compose this LLM logic in code." They are not deployed services, runtimes, or platforms.

## Where DIAL overlaps

The overlap is minimal and complementary. DIAL answers a different question: "where does that code run, how does it reach models, and how is it stored, governed, and served." DIAL provides the [Unified API](../0.what-is-dial.md) the framework calls and the application server that hosts the framework code.

## Where DIAL differs

DIAL is the platform and runtime; a framework is the in-process toolkit:

- **Runtime.** A DIAL Custom App or Code App is a deployed, managed service. A framework has no runtime of its own — it runs wherever you host it.
- **Model access.** Inside a DIAL app, framework code reaches any provider through the Unified API and adapters, instead of wiring each provider SDK directly.
- **Platform services.** Storage, access control, sharing, publications, observability, and a Chat UI surround the app. Frameworks provide none of these out of the box.

In practice you use a framework **within** DIAL, not instead of it.

## Capability comparison

| Capability | Frameworks | DIAL |
|---|---|---|
| Compose chains / agents / RAG in code | Yes | Via apps that use a framework |
| Deployed runtime / app server | No | Yes |
| Unified API across providers | No | Yes |
| Resource storage with access control | No | Yes |
| Sharing, publications, review | No | Yes |
| End-user Chat UI | No | Yes |
| Runs inside your process as a library | Yes | No (DIAL hosts the process) |

## When to use which — and together

This is not an either/or choice. Use a **framework** to write the in-process logic of an application. Use **DIAL** to host that application, give it governed access to every model, and operate it in production.

A typical pattern: build a Custom App with the [DIAL SDK](../../../building-with-dial/adapters/0.index.md), use LangChain or LlamaIndex inside it for chaining and retrieval, and reach all models through the Unified API. The framework gives you composition; DIAL gives you the platform around it.
