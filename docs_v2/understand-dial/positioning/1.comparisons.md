# Comparisons

DIAL overlaps with several categories of AI tooling, and evaluators often ask how it differs from a gateway, a low-code builder, a framework, or a vendor studio. This page orients you across those four categories and links to a detailed comparison for each. It assumes you have read [What is DIAL](./0.what-is-dial.md).

## How to read these comparisons

DIAL is designed to **complement, not replace** the rest of an AI stack. Each comparison is honest about overlap: where a tool does something DIAL also does, where it does something DIAL does not, and where the two are best used together. None of these pages argues that DIAL is categorically "better" — they explain *fit*.

A useful lens is which slice of the stack a tool occupies. Most tools do one of four things well — act as a gateway, build apps, govern access, or provide an end-user UI. DIAL's distinguishing trait is that it integrates all four as one open-source, vendor-agnostic platform.

## The four categories

| Category | Examples | DIAL's relationship |
|---|---|---|
| **AI gateways** | LiteLLM, Portkey, Kong AI Gateway | Overlap on the gateway slice — DIAL Core and adapters route across providers — but DIAL adds an app server, storage, governance, and a UI. |
| **App builders** | Dify, Flowise, n8n | Overlap with Quick Apps for visual/no-code assembly; DIAL adds an API-first substrate, governance, and a production UI. |
| **App frameworks** | LangChain, LlamaIndex, Semantic Kernel | Pure complement — these libraries run *inside* a DIAL application. |
| **AI studios** | Azure AI Foundry, Google AI Studio | Closest in breadth, but single-vendor and cloud-bound; DIAL is vendor-agnostic, open-source, and self-hostable. |
