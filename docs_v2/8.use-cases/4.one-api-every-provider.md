# One API for every model provider

The model you choose today is unlikely to be the model you want in a year, and the code you write against one provider's SDK does not move to another's. This page explains why a single, stable API across providers matters, how DIAL's Unified API delivers it, and what you give up and gain. It is for architects and evaluators weighing provider strategy, and for developers who will write against the result.

## Why this matters

Every model provider has its own SDK, authentication, request shape, and quirks. Write directly against one, and you have coupled your application to that provider: switching means rewriting integration code, retesting, and redeploying every app that touched it. Multiply that across a dozen applications and the cost of changing your mind becomes the reason you never do.

The field moves too fast for that to be acceptable. A better model ships, a provider changes pricing, a region requirement forces a different vendor — and an organization that coded against a single provider's SDK cannot react without a migration project. The value of one API across providers is the freedom to change models without changing applications.

## What you can do with DIAL

DIAL exposes the [Unified API](../2.understand-dial/2.architecture/4.unified-api-overview.md): one OpenAI-compatible interface that every model speaks, regardless of who built it. Your application talks to DIAL Core; Core talks to the provider through an [adapter](../3.building-with-dial/3.adapters/0.index.md) that translates the Unified API to and from that provider's native format.

```text
Your apps ──▶ Unified API ──▶ DIAL Core ──▶ adapter ──▶ provider
                                              adapter ──▶ provider
                                              adapter ──▶ provider
```

Because the interface is stable, swapping a model is a configuration change, not a code change. The same property lets Core route between models — by cost, by availability, or by rule — without applications knowing which model served a given request. New providers join the catalog by adding an adapter, and applications written months earlier reach them unchanged.

## How you'd build it

- **Understand the interface.** [Unified API overview](../2.understand-dial/2.architecture/4.unified-api-overview.md) explains the single contract every model honors.
- **Know your options.** [Supported providers](../3.building-with-dial/3.adapters/2.supported-providers.md) lists the models reachable today.
- **Connect a provider.** [Adapters](../3.building-with-dial/3.adapters/0.index.md) covers how a provider is attached, and [build a custom adapter](../3.building-with-dial/3.adapters/1.tutorial-custom-adapter.md) handles anything not in the catalog.
- **Route between models.** [Routes configuration](../4.operating-dial/4.configuration/1.core/2.config-json/6.routes.md) defines how Core selects a model per request.

## Why DIAL over the alternatives

A thin client library that abstracts a few providers solves part of this, but it lives in your application and still leaves each app to manage keys, limits, and observability. DIAL puts the abstraction in a gateway, so provider independence comes with central access control, cost limits, and audit for free — at the cost of running that gateway. This is the gateway category, and the [DIAL vs AI gateways](../2.understand-dial/1.positioning/3.comparisons/1.dial-vs-ai-gateways.md) comparison covers where DIAL sits within it.

Provider independence is never total: a feature unique to one provider is still unique, and the Unified API exposes common ground plus documented extensions. The win is that the common case — send a prompt, get a completion — is identical everywhere.
