# Org-wide AI assistant

Most organizations want to give every employee access to capable language models without handing each team a separate vendor account, an ungoverned API key, and a private bill. This page explains why a central AI assistant is worth building on DIAL, how DIAL delivers it, and the trade-offs against simpler alternatives. It is for decision-makers weighing the approach and for the team that would stand it up.

## Why this matters

The shadow-IT version of enterprise AI is a dozen teams each subscribing to a different chat product, pasting confidential data into tools nobody approved, and producing usage nobody can see. The risk is not hypothetical: data leaves the organization, costs are invisible until the invoice arrives, and there is no record of who asked what.

A central assistant inverts this. One interface, one identity system, one place where access rules and spending limits apply. Employees get a familiar chat experience backed by whichever models the organization approves, and the platform team keeps a single control point for security, cost, and audit.

The hard part is not the chat window. It is making one interface serve many models, many teams, and many compliance constraints at once, uniformly, without a bespoke gateway for each.

## What you can do with DIAL

DIAL is built for exactly this control point. [DIAL Chat](../2.understand-dial/2.architecture/1.architecture-highlights.md) gives employees the assistant interface, and [DIAL Core](../2.understand-dial/2.architecture/1.architecture-highlights.md) sits behind it as the gateway that every request passes through. Because all traffic flows through Core, access control, usage limits, and observability apply the same way regardless of which model or application a user picks.

```text
Employees ──▶ DIAL Chat ──▶ DIAL Core ──▶ approved models
                              │
                       access control
                       usage limits
                       audit + observability
```

Identity comes from your existing provider — the organization's single sign-on — so there is no separate user directory to maintain. Roles map groups of users to the models, applications, and quotas they are allowed. Add a new approved model once, and every employee in the right role can use it through the same interface.

## How you'd build it

The assembly is mostly configuration rather than code:

- **Stand up Core and Chat.** Start from a [local Docker Compose setup](../4.operating-dial/1.local-setup/0.index.md) to prove the flow, then move to a cloud deployment.
- **Connect identity.** Configure single sign-on through your [identity provider](../4.operating-dial/5.auth-and-access-control/0.index.md) so employees sign in with corporate credentials.
- **Define access with roles.** Use [authentication and access control](../2.understand-dial/4.security-and-governance/1.authentication-and-access-control.md) to decide which roles see which models and applications.
- **Set guardrails.** Apply [usage limits and cost control](../2.understand-dial/4.security-and-governance/3.usage-limits-and-cost-control.md) so no team or key can run away with the budget.
- **Approve models.** Attach the providers you trust through [adapters](../3.building-with-dial/3.adapters/0.index.md); the [supported providers](../3.building-with-dial/3.adapters/2.supported-providers.md) list shows what is available.
- **Hand it to users.** The [Chat User Guide](../6.chat-user-guide/0.index.md) is the manual employees follow day to day.

## Why DIAL over the alternatives

A single-vendor chat product is faster to switch on but locks you to one provider's models and gives you little control over data handling or cost attribution. Building your own gateway gives control but means writing and maintaining the access, routing, and audit layers yourself. DIAL is the assembled middle path: a governed gateway and a finished chat interface, with the trade-off that you run the platform rather than rent it. The [DIAL vs AI gateways](../2.understand-dial/1.positioning/3.comparisons/1.dial-vs-ai-gateways.md) comparison covers the gateway boundary in detail.

This pattern is worth it when you have multiple teams, more than one model worth offering, and any requirement to control data or spend. For a single small team with one model, the overhead may not pay off yet.
