# Load balancer configuration

The DIAL Core load balancer distributes requests across model deployments, regions, and cloud subscriptions. This reference documents the `upstream` parameters and the algorithms they produce. It is for DevOps engineers configuring DIAL Core.

Load balancing prevents bottlenecks, improves fault tolerance, and keeps costs under control — for example, by prioritizing provisioned-throughput (PTU) deployments before per-token-cost options.

## Upstream parameters

Load balancing is configured in DIAL Core [dynamic settings](./core/config-json/1.models.md) by defining `upstreams` on a model. These parameters can be changed without redeploying DIAL Core.

| Parameter | Required | Default | Description |
|---|---|---|---|
| `endpoint` | Yes | — | The model endpoint URL. |
| `key` | No | — | The model API key. |
| `weight` | No | `1` | The endpoint capacity as a share of the total traffic. A value of zero or below disables the endpoint. |
| `tier` | No | `0` | The endpoint group. Lower tiers receive traffic first; `0` is the highest priority. |

Minimal example:

```json
{
  "models": {
    "chat-gpt-35-turbo": {
      "upstreams": [
        {
          "endpoint": "http://0.0.0.0:7001",
          "key": "modelKey",
          "weight": 1,
          "tier": 0
        }
      ]
    }
  }
}
```

**Note**
> In a cluster, DIAL Core nodes share endpoint state asynchronously. This can cause minor inconsistency in how requests are routed.

## Standard load balancer

The standard load balancer uses round-robin distribution. Each request goes to the next endpoint in the list, cycling back to the first after the last. This suits resources with similar capabilities handling comparable workloads.

```json
{
  "models": {
    "chat-gpt-35-turbo": {
      "upstreams": [
        { "endpoint": "https://hostname1/openai/deployments/gpt-4-32k-0613/chat/completions" },
        { "endpoint": "https://hostname2/openai/deployments/gpt-4-32k-0613/chat/completions" },
        { "endpoint": "https://hostname3/openai/deployments/gpt-4-32k-0613/chat/completions" }
      ]
    }
  }
}
```

Requests are distributed in the order specified. Each endpoint defaults to `weight` `1` and `tier` `0`.

## Advanced load balancer

Combine `tier` and `weight` to define a more complex algorithm.

### Tiers

Use `tier` to group endpoints. Requests route to the lowest-tier group first; higher tiers absorb load during an outage or when limits are reached. For example, requests reach `"tier": 0` endpoints before `"tier": 1`.

```json
{
  "models": {
    "chat-gpt-35-turbo": {
      "upstreams": [
        { "endpoint": "https://hostname1/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 0 },
        { "endpoint": "https://hostname2/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 0 },
        { "endpoint": "https://hostname3/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 1 }
      ]
    }
  }
}
```

### Weights

Use `weight` within a tier to allocate a share of traffic. Endpoints receive requests with probability proportional to their weights. A positive value sets capacity; zero or below disables the endpoint from routing.

**Note**
> `weight` only takes effect in combination with `tier`.

```json
{
  "models": {
    "chat-gpt-35-turbo": {
      "upstreams": [
        { "endpoint": "https://hostname1/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 0, "weight": 3 },
        { "endpoint": "https://hostname2/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 0, "weight": 2 },
        { "endpoint": "https://hostname3/openai/deployments/gpt-4-32k-0613/chat/completions", "tier": 1, "weight": 1 }
      ]
    }
  }
}
```

Requests first reach `tier 0`, split between `hostname1` and `hostname2` proportional to their weights. If `tier 0` is unavailable or at capacity, traffic moves to `tier 1`.

### Fallbacks

Fallback logic keeps the system resilient when upstreams fail. It avoids repeated retries to the same endpoint and redirects requests to alternatives.

- **Endpoint selection** — when lower-tier endpoints become unavailable, traffic moves to the next-higher tier.
- **Retry limitation** — the `maxRetryAttempts` parameter caps retries to a single deployment. The default is `5` for language models and `1` for applications.
- **Conditional fallback** — if all upstreams are unavailable, the balancer may retry an endpoint that previously returned `500` (Internal Server Error) or `429` (Too Many Requests), provided it meets the predefined criteria.
