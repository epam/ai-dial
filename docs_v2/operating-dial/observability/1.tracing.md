# Configure tracing with OpenTelemetry

This guide enables distributed tracing across DIAL so you can follow a single request from DIAL Core through an adapter, interceptor, and application. It is for DevOps engineers who already run DIAL and have an OTLP-compatible collector or tracing backend. For the conceptual model, see [Observability](index).

## Prerequisites

- A running DIAL deployment (Docker Compose or Kubernetes)
- An OpenTelemetry Collector, or a tracing backend that accepts OTLP (such as Jaeger, Grafana Tempo, or Zipkin)
- Access to set environment variables on each DIAL component you want to trace

## How tracing flows across components

DIAL Core assigns each request a trace context and propagates it to downstream components. Core emits the following identifiers, which Analytics Realtime also records, letting you join a trace to its usage data:

| Identifier | Meaning |
|------------|---------|
| `trace_id` | The OpenTelemetry trace ID for the whole request |
| `core_span_id` | The span ID generated by DIAL Core for a call |
| `core_parent_span_id` | The span ID of the caller that invoked `core_span_id` |
| `execution_path` | A `/`-separated call stack, for example `app1/app2/model1` |

A request that fans out across applications and models produces one trace with nested spans, so you can see which leg was slow or failed.

## Configure DIAL Core

DIAL Core reads the standard OpenTelemetry environment variables. Set the OTLP endpoint and enable the trace exporter:

```bash
OTEL_SERVICE_NAME=dial-core
OTEL_TRACES_EXPORTER=otlp
OTEL_EXPORTER_OTLP_ENDPOINT=http://<YOUR_COLLECTOR_HOST>:4317
```

:::note
DIAL Core is a Java service. These variables are read by the OpenTelemetry Java agent, not by `config.json`. How the agent jar is attached to the JVM is set at the image or deployment level — confirm the attachment mechanism for your deployment before relying on these variables.
:::

## Configure DIAL Chat

DIAL Chat is a Node.js service. Point it at the same collector:

```bash
OTEL_SERVICE_NAME=dial-chat
OTEL_TRACES_EXPORTER=otlp
OTEL_EXPORTER_OTLP_ENDPOINT=http://<YOUR_COLLECTOR_HOST>:4317
```

Set `OTEL_SDK_DISABLED=true` to turn the Node.js instrumentation off entirely.

## Configure adapters and applications

Adapters and applications built on the [DIAL SDK](../../building-with-dial/developer-tools/sdk-reference/5.telemetry.md) are Python services. They read these variables:

```bash
OTEL_RESOURCE_ATTRIBUTES=service.name=<YOUR_SERVICE_NAME>
OTEL_TRACES_EXPORTER=otlp
OTEL_EXPORTER_OTLP_ENDPOINT=http://<YOUR_COLLECTOR_HOST>:4317
OTEL_PYTHON_LOG_CORRELATION=true
```

When tracing is enabled, the SDK instruments outgoing HTTP calls (`requests`, `aiohttp`, `httpx`, `urllib`) and FastAPI request handling, so adapter-to-provider calls appear as child spans. `OTEL_PYTHON_FASTAPI_EXCLUDED_URLS` excludes paths such as health checks from tracing.

For the full SDK telemetry surface — `TelemetryConfig`, programmatic setup, and every supported variable — see the [SDK telemetry reference](../../building-with-dial/developer-tools/sdk-reference/5.telemetry.md).

## Correlate traces with logs

Set `OTEL_PYTHON_LOG_CORRELATION=true` on SDK-based components to inject `trace_id` and `span_id` into log lines. With correlation on, you can pivot from a slow trace to the exact log records emitted during that request.

## Result

Each request produces a single trace spanning every configured component. Open your tracing backend and search by `trace_id` to see the span tree, including adapter calls to upstream providers.

:::warning
DIAL does not expose its own trace-sampling configuration. Sampling is governed by the standard OpenTelemetry agent and SDK environment variables (for example `OTEL_TRACES_SAMPLER`), which are general OTEL behavior rather than a DIAL feature. Validate sampling against your collector and backend.
:::
