# What are adapters

Adapters are DIAL components that translate an external AI provider's API into the DIAL Unified API. If you build applications on DIAL, adapters let you consume models from Azure OpenAI, AWS Bedrock, Google Vertex AI, and other providers through a single, consistent interface — without writing provider-specific code.

## Why adapters exist

Enterprise deployments typically consume models from multiple providers. Each provider exposes a different API: different authentication, different request formats, different streaming protocols. Without a normalization layer, every client application must handle each provider individually. Changes to a provider's API ripple through every consumer.

Adapters eliminate this coupling. They sit between DIAL Core and the provider, translating requests and responses in both directions. Applications, interceptors, and DIAL Chat interact with all models identically through the Unified API, regardless of which provider serves the model.

## How adapters work

When a client sends a chat completion request to DIAL Core, Core looks up the target model's deployment in `config.json` and forwards the request to the adapter endpoint configured for that model. The adapter translates the request into the provider's native format, sends it upstream, then translates the response back into the Unified API format before returning it to Core.

```
Request:   Client → DIAL Core → Adapter → Provider API
Response:  Provider API → Adapter → DIAL Core → Client
```

Each adapter runs in its own container. A single adapter instance can serve multiple models from the same provider. Adapters support streaming, embeddings, and all Unified API features that the underlying provider supports.

## Adapters vs. applications

Adapters and Custom Apps are built with the same [DIAL SDK](https://github.com/epam/ai-dial-sdk). Both subclass `ChatCompletion`, both register with `DIALApp`, and both expose endpoints that conform to the Unified API. The difference is intent:

| | Adapter | Application |
|---|---|---|
| **Purpose** | Proxy requests to an external AI provider | Implement business logic |
| **Input** | Unified API request | Unified API request |
| **Processing** | Translate format, forward to provider, translate response | Execute custom logic, optionally call other models |
| **SDK** | [DIAL SDK](https://github.com/epam/ai-dial-sdk) (`aidial-sdk`) | [DIAL SDK](https://github.com/epam/ai-dial-sdk) (`aidial-sdk`) |

Because they share the same SDK, building a custom adapter requires the same skills as building a Custom App. The [DIAL SDK reference](../5.developer-tools/1.sdk-reference/0.index.md) covers the full API surface for both.

## Official adapters

DIAL ships three adapters that cover the major AI model providers:

| Adapter | Providers | Repository |
|---|---|---|
| **OpenAI Adapter** | Azure OpenAI, OpenAI, Mistral, Databricks, vLLM, Ollama, and other OpenAI-compatible APIs | [ai-dial-adapter-openai](https://github.com/epam/ai-dial-adapter-openai) |
| **Bedrock Adapter** | AWS Bedrock (Anthropic Claude, Amazon Titan, Meta Llama, Stability AI, Cohere, AI21, DeepSeek) | [ai-dial-adapter-bedrock](https://github.com/epam/ai-dial-adapter-bedrock) |
| **Vertex AI Adapter** | Google Vertex AI (Gemini, PaLM, Imagen, Anthropic Claude on Vertex) | [ai-dial-adapter-vertexai](https://github.com/epam/ai-dial-adapter-vertexai) |

For the full model compatibility table, see [Supported providers](2.supported-providers.md).

## Building a custom adapter

Build a custom adapter when you need to integrate a provider that the official adapters do not cover — for example, a proprietary model behind a non-standard API, or a fine-tuned model hosted on internal infrastructure.

A custom adapter is a Python application that subclasses `ChatCompletion` from the DIAL SDK, translates requests to the target API format, and translates responses back. The [Tutorial: custom adapter](1.tutorial-custom-adapter.md) walks through the process end to end.

## Configuration

Deploying an adapter in DIAL requires two pieces of configuration:

1. **Adapter container environment variables** — credentials, connection settings, and feature flags for the adapter itself. See the [adapter configuration reference](../../4.operating-dial/4.configuration/3.adapter-configuration.md) for all supported variables.

2. **Model routing in DIAL Core** — each model deployment in `config.json` specifies the adapter endpoint and upstream settings. See the [config.json models reference](../../4.operating-dial/4.configuration/1.core/2.config-json/1.models.md) for the schema.
