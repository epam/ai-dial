# Evaluate RAG quality with DIAL RAG Eval

This guide shows how to measure retrieval and generation quality of a RAG application using the [DIAL RAG Eval](https://github.com/epam/ai-dial-rag-eval) library. It is aimed at developers who have a working RAG application and want to quantify its performance. You should be familiar with Python, pandas DataFrames, and basic RAG concepts.

## Prerequisites

- Python 3.11 or later
- A RAG application that produces answers with retrieved context chunks
- Ground truth data for your test questions (or use the sample data in this guide)
- For generation metrics: access to an LLM through DIAL or a LangChain-compatible provider

## Install the library

Install DIAL RAG Eval from PyPI:

```bash
pip install aidial-rag-eval==0.6.0
```

Generation metrics require the spaCy English language model for sentence tokenization. Download it after installation:

```bash
python -m spacy download en_core_web_sm
```

## Prepare evaluation data

DIAL RAG Eval takes two pandas DataFrames as input: one with ground truth data and one with collected answers from your RAG application.

### Ground truth DataFrame

The ground truth contains the expected answers and key facts for each question.

| Column | Type | Description |
|---|---|---|
| `question` | `str` | The question being evaluated |
| `documents` | `list[str]` | Document identifiers (used as join key with answers) |
| `facts` | `list[str]` | Ground truth facts—the key pieces of information a correct answer must contain |
| `answer` | `str` | The expected answer text |

### Collected answers DataFrame

The collected answers contain the actual responses produced by your RAG application.

| Column | Type | Description |
|---|---|---|
| `question` | `str` | The question (must match ground truth) |
| `documents` | `list[str]` | Document identifiers (must match ground truth) |
| `context` | `list[str]` | Retrieved context chunks |
| `answer` | `str` | The generated answer |

The two DataFrames are joined on the `(documents, question)` pair.

### Example data

```python
import pandas as pd

df_ground_truth = pd.DataFrame([
    {
        "question": "When was Apple Inc. established and who was the CEO in 2020?",
        "documents": ["apple.pdf"],
        "facts": [
            "Apple Inc. was established on April 1, 1976.",
            "The CEO of Apple in 2020 was Tim Cook.",
        ],
        "answer": "Apple Inc. was established on April 1, 1976. The CEO in 2020 was Tim Cook.",
    },
])

df_answer = pd.DataFrame([
    {
        "question": "When was Apple Inc. established and who was the CEO in 2020?",
        "documents": ["apple.pdf"],
        "context": [
            "Apple Inc. is an American multinational technology company founded on April 1, 1976.",
            "Tim Cook has been the CEO of Apple Inc. since August 2011.",
        ],
        "answer": "Apple Inc. was established on April 1, 1976. Tim Cook was the CEO in 2020.",
    },
])
```

## Run retrieval metrics

Retrieval metrics do not require an LLM. They measure how well the retrieval step finds the right information by matching ground truth facts to retrieved context chunks.

```python
from aidial_rag_eval import create_rag_eval_metrics_report

df_metrics = create_rag_eval_metrics_report(df_ground_truth, df_answer)

print(df_metrics[["recall", "precision", "mrr", "f1"]])
```

| Metric | What it measures |
|---|---|
| `recall` | Fraction of ground truth facts found in any retrieved chunk |
| `precision` | Fraction of retrieved chunks that contain at least one ground truth fact |
| `mrr` | Mean Reciprocal Rank—average inverse rank of the first relevant chunk per fact |
| `f1` | Harmonic mean of precision and recall |

The default matcher uses exact substring matching—a fact is considered found if it appears as a substring in any context chunk.

## Run generation metrics

Generation metrics use an LLM judge to evaluate entailment between text pairs. You need a LangChain-compatible chat model.

### Configure the LLM judge

To use a model through a DIAL instance:

```python
from langchain_openai import AzureChatOpenAI

llm = AzureChatOpenAI(
    azure_endpoint="<YOUR_DIAL_URL>",
    api_key="<YOUR_DIAL_API_KEY>",
    model="gpt-5-mini",
    api_version="2024-12-01-preview",
)
```

**Tip**
> Generation metrics are token-intensive. For the best balance of quality and cost, use one of these models: `gemini-2.5-flash-lite`, `gemini-3.1-flash-lite`, `gpt-5-mini` (low reasoning effort), or `gpt-5-nano` (low reasoning effort).

### Select metric binds

Each metric bind defines a premise-hypothesis pair for entailment scoring:

| Constant | Prefix | What it measures |
|---|---|---|
| `CONTEXT_TO_ANSWER_INFERENCE` | `ctx_ans_` | Faithfulness—can the answer be derived from the context? |
| `ANSWER_TO_GROUND_TRUTH_INFERENCE` | `ans_gt_` | Coverage—does the answer entail the expected answer? |
| `GROUND_TRUTH_TO_ANSWER_INFERENCE` | `gt_ans_` | Correctness—does the expected answer entail the generated answer? |
| `ANSWER_TO_FACTS_INFERENCE` | `ans_fct_` | Fact coverage—does the answer cover all expected facts? |
| `FACTS_TO_ANSWER_INFERENCE` | `fct_ans_` | Factual grounding—are the answer's claims supported by expected facts? |
| `ANSWER_REFUSAL` | `answer_` | Whether the model refused to answer |
| `GROUND_TRUTH_REFUSAL` | `ground_truth_` | Whether the ground truth is a refusal |

### Run the evaluation

```python
from aidial_rag_eval import create_rag_eval_metrics_report
from aidial_rag_eval.metric_binds import (
    CONTEXT_TO_ANSWER_INFERENCE,
    ANSWER_TO_GROUND_TRUTH_INFERENCE,
    GROUND_TRUTH_TO_ANSWER_INFERENCE,
    ANSWER_TO_FACTS_INFERENCE,
    FACTS_TO_ANSWER_INFERENCE,
)

df_metrics = create_rag_eval_metrics_report(
    df_ground_truth,
    df_answer,
    llm=llm,
    metric_binds=[
        CONTEXT_TO_ANSWER_INFERENCE,
        ANSWER_TO_GROUND_TRUTH_INFERENCE,
        GROUND_TRUTH_TO_ANSWER_INFERENCE,
        ANSWER_TO_FACTS_INFERENCE,
        FACTS_TO_ANSWER_INFERENCE,
    ],
)

print(df_metrics[[
    "recall", "precision", "mrr", "f1",
    "ctx_ans_inference", "ans_gt_inference", "gt_ans_inference",
    "ans_fct_inference", "fct_ans_inference",
]])
```

### Use the file-based API

For larger datasets stored as Parquet files, use the `evaluate` function:

```python
from aidial_rag_eval import evaluate
from aidial_rag_eval.metric_binds import CONTEXT_TO_ANSWER_INFERENCE

result = evaluate(
    ground_truth="./ground_truth.parquet",
    answers="./answers.parquet",
    dest="./metrics.parquet",
    llm=llm,
    metric_binds=[CONTEXT_TO_ANSWER_INFERENCE],
    max_concurrency=8,
)
```

This reads input from Parquet, writes results to a Parquet file at `dest`, and returns a `Dataset` object with aggregated statistics.

## Interpret results

### Retrieval metrics

- **High recall, low precision** — the retriever finds the right information but returns too many irrelevant chunks. Reduce the top-k parameter or improve the embedding model.
- **Low recall, high precision** — the retrieved chunks are relevant but some ground truth facts are missing. Increase top-k, reduce chunk size, or improve the indexing strategy.
- **Low MRR** — relevant chunks appear late in the ranked list. Adjust the ranking model or retrieval algorithm.

### Generation metrics

All inference scores range from 0.0 to 1.0. A score of 1.0 means full entailment.

- **Low `ctx_ans_inference`** — the model generates claims not supported by the context (hallucination). Tighten the system prompt to restrict answers to the retrieved context.
- **Low `ans_gt_inference`** — the generated answer does not cover the expected answer. The model may be omitting information. Check whether the relevant context is being retrieved (look at recall first).
- **Low `ans_fct_inference`** — the answer misses specific ground truth facts. Similar diagnosis to `ans_gt_inference`, but at the fact level.
- **`answer_refusal` = 1.0** — the model refused to answer. Check whether the context contains the needed information and whether the system prompt is too restrictive.

## Result

You now have retrieval and generation quality metrics for your RAG application. Use these metrics to identify weaknesses and guide improvements.
