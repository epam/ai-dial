# Manage users and their access

This guide explains how user identity works in DIAL and how administrators grant, change, and remove
a user's access. It is for administrators who govern who can use a DIAL deployment and what they can
reach. It assumes you have access to DIAL Admin and to your organization's identity provider (IdP).

## DIAL does not store user accounts

DIAL has no user directory and no per-user create, edit, or delete screen. Identity is delegated to
your **identity provider** over OpenID Connect. Each interactive user signs in through the IdP, and
DIAL trusts the resulting JWT. Service-to-service callers authenticate with [API keys](access-management/keys.md)
instead.

Because of this, "managing a user" in DIAL means managing the **roles** their identity maps to, not
editing an account record. The IdP owns the lifecycle of the account; DIAL owns what that account can
do once it arrives. For the model behind this, see
[Authentication and access control](../understand-dial/security-and-governance/1.authentication-and-access-control.md).

## How an identity becomes a set of permissions

1. A user authenticates with the IdP and receives a JWT containing claims, including group or role
   claims.
2. DIAL Core maps those claims to one or more DIAL [roles](access-management/roles.md).
3. The roles determine which models, applications, tool sets, and routes the user can access, and the
   [usage and cost limits](9.usage-limits-and-cost-control.md) that apply.

The claim-to-role mapping is configured per identity provider. See
[JWT configuration](../operating-dial/auth-and-access-control/2.jwt.md) and your provider's guide
under [SSO / IdP setup](../operating-dial/auth-and-access-control/sso-idp/0.index.md).

## Grant a user access

1. Confirm the user can sign in through the IdP and that their token carries the expected group or
   role claim. Group membership is managed in the IdP, not in DIAL.
2. In DIAL Admin, open **Access management > Roles** and choose or
   [create a role](access-management/roles.md) for that claim.
3. On the role's **Entities** tab, add the models, applications, tool sets, and routes the user should
   reach, and set their limits.
4. Confirm DIAL Core maps the user's claim to that role (see
   [JWT configuration](../operating-dial/auth-and-access-control/2.jwt.md)).

**Result**

> When the user next signs in, their token resolves to the role, and they gain exactly the access the
> role grants.

## Provision a service identity

For automated callers and integrations, issue an [API key](access-management/keys.md) instead of an
interactive login:

1. Open **Access management > Keys** and create a key with an expiration that enforces rotation.
2. Assign the roles the integration needs.
3. Deliver the secret to the integration owner through a secure channel.

## Grant or revoke administrator rights

Administrators are not assigned in the panel. They are defined in DIAL Core's static settings, in the
`access.admin.rules` block, which matches against token claims. To grant or revoke admin rights, edit
that configuration — see [Access Control settings](../operating-dial/configuration/core/settings-json/7.access.md).
Read-only administrators are restricted through roles; see
[Read-only access](0.index.md#read-only-access).

## Offboard a user

Because DIAL trusts the IdP, removing a user's access starts there:

1. Disable the account or remove the group claim in the IdP. The user can no longer obtain a valid
   token, and access stops at the next sign-in.
2. Revoke any [API keys](access-management/keys.md) the user held that should no longer be valid.
3. Review the user's shared resources and publications in
   [Publications and review](6.publications-and-review.md) and reassign or unpublish as needed.
