# Deploy DIAL on a generic Kubernetes cluster

Deploy the full DIAL stack to any Kubernetes cluster that is not tied to a specific cloud provider — a self-managed cluster, an on-premises cluster, or a local distribution such as minikube or kind. This guide assumes you are familiar with Helm and `kubectl`. For provider-managed clusters, use the [AWS](aws-deployment), [Azure](azure-deployment), or [GCP](gcp-deployment) guides instead, which integrate managed storage and identity.

## Prerequisites

- A Kubernetes cluster, version 1.23 or higher, with `kubectl` and Helm 3.8.0 or higher configured against it
- A default `StorageClass` for persistent volumes (DIAL Core stores files on a persistent volume rather than cloud object storage)
- An ingress controller, if you want to expose DIAL outside the cluster
- An endpoint and key for at least one model provider reachable from the cluster
- The DIAL Helm repository added:

  ```bash
  helm repo add dial https://charts.dialx.ai
  helm repo update
  ```

## Step 1: Create a values file

Start from the upstream [generic `simple` example](https://github.com/epam/ai-dial-helm/tree/main/charts/dial/examples/generic/simple), which deploys DIAL with a persistent-volume file store and a single model adapter. The required values are:

```yaml
core:
  # Files are stored on a persistent volume instead of cloud object storage
  persistence:
    enabled: true
    size: 10Gi
    # storageClass: <STORAGE_CLASS>   # omit to use the cluster default

openai:
  env:
    aidial.openai.endpoint: <MODEL_ENDPOINT>
    aidial.openai.key: <MODEL_KEY>

ingress:
  enabled: true
  hosts:
    - host: <DOMAIN>
```

Replace the placeholders:

- `<STORAGE_CLASS>` — a specific `StorageClass`, or omit the line to use the cluster default
- `<MODEL_ENDPOINT>` — the endpoint of your model provider
- `<MODEL_KEY>` — the API key for the model provider
- `<DOMAIN>` — the host for ingress, for example `dial.example.com`

Generate the DIAL API key and encryption key before installing and supply them as values or [Kubernetes secrets](../configuration/core/settings-json/encryption).

## Step 2: Install the chart

```bash
helm install dial dial/dial \
  -f values.yaml \
  --version <CHART_VERSION> \
  --namespace dial \
  --create-namespace
```

Pin `<CHART_VERSION>` to a released chart version.

## Step 3: Verify the deployment

Check that the pods are running:

```bash
kubectl get pods -n dial
```

If you did not configure ingress, port-forward DIAL Core to reach the Unified API:

```bash
kubectl port-forward svc/dial-core 8080:8080 -n dial
curl http://localhost:8080/openai/deployments \
  -H "Api-Key: <YOUR_DIAL_API_KEY>"
```

## Result

DIAL Core, DIAL Chat, and Redis run on your cluster, with files persisted to a Kubernetes persistent volume and models served from your configured provider.

## Production architecture

The generic example uses a persistent volume and a single model endpoint, which suits evaluation and on-premises installs. For production resilience, address the same concerns the managed-cloud guides cover:

- **Stateless scaling** — DIAL services are stateless. Run at least two copies of each service across nodes and configure a `HorizontalPodAutoscaler` for DIAL Core based on CPU usage.
- **Storage durability** — a single persistent volume is a single point of failure. Back the volume with replicated or networked storage, and schedule snapshots or backups of the file store.
- **Cache** — Redis is a cache and can be restarted safely, but a production deployment should run a highly available Redis rather than the in-cluster single instance.
- **Model resilience** — define multiple model endpoints so DIAL fails over automatically when one is unavailable.
- **Secrets** — store the API key, encryption key, and model credentials in your secret manager rather than in the values file.
