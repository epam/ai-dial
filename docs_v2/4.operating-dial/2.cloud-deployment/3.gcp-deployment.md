# Deploy DIAL on GCP

Deploy the full DIAL stack to Google Kubernetes Engine (GKE) using the umbrella Helm chart, with Cloud Storage for files and Vertex AI as the model provider. This guide assumes you operate a GKE cluster and are familiar with Helm and `kubectl`. For the chart model and infrastructure mapping, read the [Helm chart overview](index) first.

## Prerequisites

- A GKE cluster, version 1.24 or higher, with `kubectl` and Helm 3.8.0 or higher configured against it
- The [Ingress-NGINX controller](https://kubernetes.github.io/ingress-nginx/) and, optionally, [cert-manager](https://cert-manager.io/) installed in the cluster
- [Workload Identity Federation for GKE](https://cloud.google.com/kubernetes-engine/docs/how-to/workload-identity) configured
- A Cloud Storage bucket for DIAL file storage
- Vertex AI enabled in your project with at least one model available
- A domain and a cert-manager `ClusterIssuer` for TLS
- The DIAL Helm repository added:

  ```bash
  helm repo add dial https://charts.dialx.ai
  helm repo update
  ```

## Step 1: Create a values file

Start from the upstream [GCP `simple` example](https://github.com/epam/ai-dial-helm/tree/main/charts/dial/examples/gcp/simple) and fill in your environment. The required values are:

```yaml
core:
  # Workload Identity Federation service account for Core
  serviceAccount:
    annotations:
      iam.gke.io/gcp-service-account: <GCP_CORE_SERVICE_ACCOUNT>
  env:
    aidial.storage.bucket: <GCP_CORE_STORAGE_BUCKET_NAME>

vertexai:
  serviceAccount:
    annotations:
      iam.gke.io/gcp-service-account: <GCP_VERTEXAI_SERVICE_ACCOUNT>
  env:
    aidial.vertexai.project: <GCP_PROJECT_ID>
    aidial.vertexai.region: <GCP_REGION>

ingress:
  enabled: true
  hosts:
    - host: <DOMAIN>
  tls:
    - secretName: dial-tls
      hosts:
        - <DOMAIN>
  annotations:
    cert-manager.io/cluster-issuer: <CLUSTER_ISSUER>
```

Replace the placeholders:

- `<GCP_CORE_SERVICE_ACCOUNT>` — the federated service account for DIAL Core's storage access
- `<GCP_CORE_STORAGE_BUCKET_NAME>` — the Cloud Storage bucket name
- `<GCP_VERTEXAI_SERVICE_ACCOUNT>` — the service account for Vertex AI access
- `<GCP_PROJECT_ID>` — your GCP project ID
- `<GCP_REGION>` — the region where Vertex AI models are available
- `<DOMAIN>` — your domain, for example `dial.example.com`
- `<CLUSTER_ISSUER>` — the cert-manager issuer, for example `letsencrypt-production`

Generate the DIAL API key, encryption key, and Redis password before installing and supply them as values or [Kubernetes secrets](../configuration/core/settings-json/encryption).

## Step 2: Install the chart

```bash
helm install dial dial/dial \
  -f values.yaml \
  --version <CHART_VERSION> \
  --namespace dial \
  --create-namespace
```

Pin `<CHART_VERSION>` to a released chart version.

## Step 3: Verify the deployment

Check that the pods are running:

```bash
kubectl get pods -n dial
```

Once the ingress has an address and the certificate is issued, DIAL Chat is reachable at `https://<DOMAIN>/`. Confirm the Unified API responds:

```bash
curl https://<DOMAIN>/openai/deployments \
  -H "Api-Key: <YOUR_DIAL_API_KEY>"
```

## Result

DIAL Core, DIAL Chat, Redis, and the Vertex AI adapter run on your GKE cluster, with files persisted to Cloud Storage and models served from Vertex AI.

## Production architecture

The `simple` example is a starting point. For a production deployment, size the GCP services and configure redundancy as follows.

### Recommended service tiers

| GCP service | Configuration / tier |
|-------------|----------------------|
| Google Kubernetes Engine | Standard, regional cluster with zone redundancy |
| Cloud Load Balancing | Global external HTTP(S) |
| Cloud Storage | Standard, dual-region or multi-region |
| Memorystore for Redis | Standard tier, HA configuration |
| Vertex AI | Sized to capacity requirements; usage-based pricing |

Dependencies whose technology may follow corporate standards:

| Dependency | Google Cloud service |
|------------|----------------------|
| SSO | Cloud Identity |
| Firewall | Cloud Armor for WAF, plus VPC firewall rules |
| Logging and monitoring | Google Cloud Observability |
| Container registry | Artifact Registry |
| Secret storage | Secret Manager |

### Scalability

DIAL services are stateless, so they run as multiple concurrent copies. As a baseline, run two copies of each service on different nodes for high availability. The services are not resource-intensive and handle thousands of users without further scaling.

To absorb load spikes, configure autoscaling for DIAL Core based on CPU usage (for example, scale out when usage exceeds 80% of the CPU limit for one minute), and enable cluster autoscaling so new nodes are added when existing nodes lack capacity.

Scaling language models is the more common need, because a single model instance is often capped by request rate limits. With Vertex AI there are four levels of scaling, applied in order:

1. Increase the number of model instances in the same region.
2. Request a regional quota extension through the GCP console — each region has a fixed tokens-per-minute (TPM) quota per model type.
3. Deploy model instances in additional regions.
4. Use multiple GCP projects to access model endpoints.

### High availability

Achieve high availability through regional and multi-zone capabilities: run DIAL service copies in a multi-zone GKE cluster and use highly available configurations for the supporting services.

| Service | Availability SLA | Resilient to zone-wide outages |
|---------|------------------|--------------------------------|
| Google Kubernetes Engine | 99.95% with zone redundancy, otherwise 99.5% | Yes, if configured explicitly |
| Cloud Identity | 99.9%–99.99% | Yes |
| Cloud Storage | 99.95% (dual/multi-region), otherwise 99.9% | Yes |
| Memorystore for Redis | 99.9% with zone redundancy | Yes, if configured explicitly |
| Vertex AI | 99.5%–99.9% | Depends on configuration |
| Secret Manager | 99.95% | Yes |
| Artifact Registry | 99.95% | Yes (regional) |

To improve resilience beyond the Vertex AI SLA, define multiple model endpoints in different regions. If one region becomes unavailable, DIAL automatically tries the next endpoint in the list.

### Disaster recovery

DIAL has two parts with separate recovery policies: DIAL Core and the optional DIAL Analytics Realtime component.

DIAL Core consists of stateless services plus a single permanent store on Cloud Storage (Memorystore for Redis is a cache and can be safely restarted). For a region-wide failure:

- **Cloud Storage** — dual-region or multi-region buckets provide built-in cross-region redundancy. After a regional failure, review and re-establish the multi-region configuration as needed.
- **All other services** — being stateless, redeploy the full set in a secondary region and re-route the DIAL domain name to the new entry point. To meet a tight recovery time objective, keep a passive DIAL Core deployment in the secondary region and activate it on failure.
