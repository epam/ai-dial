# Configure logging

This guide sets log levels for DIAL components and ships their logs off the container with the Vector sidecar in the Helm chart. It is for DevOps engineers operating DIAL who need logs for debugging or aggregation. For the conceptual model, see [Observability](index).

## Prerequisites

- A running DIAL deployment
- For log shipping: DIAL deployed with the `dial-core` Helm chart
- A log aggregator if you want to forward logs beyond standard output

## Set log levels per component

Each component reads its own log-level variables. Supported levels run from least to most verbose: `ERROR`, `WARN`, `INFO`, `DEBUG`, `TRACE`.

| Component | Variables |
|-----------|-----------|
| DIAL Core | `AIDIAL_LOG_LEVEL`, `AIDIAL_LOG_FILE` (log file location) |
| DIAL Chat | `OTEL_LOG_LEVEL`, `OTEL_EXPORTER_OTLP_LOGS_ENDPOINT`, `OTEL_EXPORTER_OTLP_LOGS_TIMEOUT` |
| Bedrock Adapter, Vertex AI Adapter | `LOG_LEVEL` and `AIDIAL_LOG_LEVEL` |
| OpenAI Adapter, DIAL-to-DIAL Adapter | `LOG_LEVEL` |

For example, raise DIAL Core to debug logging:

```bash
AIDIAL_LOG_LEVEL=DEBUG
```

DIAL Chat can also export logs over OTLP. Point it at your collector with `OTEL_EXPORTER_OTLP_LOGS_ENDPOINT` and disable the SDK entirely with `OTEL_SDK_DISABLED=true` when needed.

## Where DIAL Core writes logs

DIAL Core writes log files to the directory set by `LOG_DIR`, which defaults to `/app/log`. Containers also stream logs to standard output and standard error, so `kubectl logs` and `docker logs` work without extra configuration.

:::note
The on-disk log format of DIAL Core (structured JSON versus plain text) is not documented in the source repositories. Verify the format your version produces before writing parsing rules in a downstream aggregator.
:::

## Ship logs with the Vector sidecar

The `dial-core` Helm chart includes an optional [Vector](https://vector.dev/) sidecar that tails Core's log files. Enable it under the `logger` key:

```yaml
logger:
  enabled: true
  image:
    repository: timberio/vector
```

By default the sidecar tails `/app/log/*.log` and prints to standard output:

```yaml
logger:
  config:
    sources:
      core_logs:
        type: file
        include: [/app/log/*.log]
    sinks:
      console:
        inputs: [core_logs]
        type: console
        target: stdout
        encoding:
          codec: text
```

## Forward logs to an aggregator

To send logs somewhere other than standard output, override `logger.config` with a Vector sink for your aggregator. Vector supports sinks for Elasticsearch, Loki, HTTP endpoints, and many others. Keep the `core_logs` source and replace or add to the `sinks` block.

The same mechanism feeds the usage-analytics pipeline: an HTTP sink pointed at the Analytics Realtime service. See [Analytics Realtime setup](analytics-realtime-setup) for that configuration.
