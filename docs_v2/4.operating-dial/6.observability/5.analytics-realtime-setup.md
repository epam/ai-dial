# Set up Analytics Realtime

This guide stands up the DIAL usage-analytics pipeline: DIAL Core logs flow through Vector to the Analytics Realtime service, which writes token, cost, and topic metrics to InfluxDB for Grafana. It is for DevOps engineers who want usage and cost visibility. For where this fits among the observability planes, see [Observability](index).

## Prerequisites

- A running DIAL deployment with the [Vector log sidecar](logging#ship-logs-with-the-vector-sidecar) available
- An InfluxDB instance (version 2 or 3)
- A deployment target for the [Analytics Realtime](https://github.com/epam/ai-dial-analytics-realtime) service

## How the pipeline works

Analytics Realtime is a separate plane from the OpenTelemetry runtime-health signals. It analyzes DIAL Core's request logs to extract usage insights without storing sensitive message content.

```mermaid
flowchart LR
  Core[DIAL Core] -->|request logs| Vector[Vector]
  Vector -->|HTTP sink| AR[Analytics Realtime]
  AR --> Influx[(InfluxDB)]
  Influx --> Grafana[Grafana]
```

The Analytics Realtime dev server listens on port `5001`. Only requests that returned HTTP 200 are processed.

## Step 1: Point Vector at Analytics Realtime

Override the Vector `logger.config` in the `dial-core` Helm chart to add an HTTP sink that forwards Core logs to the Analytics Realtime service. Keep the existing `core_logs` file source and add the sink alongside it:

```yaml
logger:
  enabled: true
  config:
    sources:
      core_logs:
        type: file
        include: [/app/log/*.log]
    sinks:
      analytics:
        inputs: [core_logs]
        type: http
        uri: http://<ANALYTICS_REALTIME_HOST>:5001/<ANALYTICS_PATH>
        encoding:
          codec: json
```

:::note
Confirm the exact sink path and request encoding the Analytics Realtime service expects against the [service repository](https://github.com/epam/ai-dial-analytics-realtime) and your DIAL Core version — these are not fixed in the chart defaults.
:::

## Step 2: Configure the InfluxDB connection

Set the connection environment variables on the Analytics Realtime service. The variables differ by InfluxDB version.

InfluxDB 2:

```bash
INFLUX_URL=http://<INFLUXDB_HOST>:8086
INFLUX_ORG=<YOUR_ORG>
INFLUX_BUCKET=<YOUR_BUCKET>
INFLUX_API_TOKEN=<YOUR_TOKEN>
```

InfluxDB 3:

```bash
INFLUX_URL=http://<INFLUXDB_HOST>:8086
INFLUX_DATABASE=<YOUR_DATABASE>
INFLUX_API_TOKEN=<YOUR_TOKEN>
```

Optional variables:

| Variable | Purpose |
|----------|---------|
| `TOPIC_MODEL` | Hugging Face model for topic classification |
| `TOPIC_EMBEDDINGS_MODEL` | Embeddings model for topic classification |
| `LOG_LEVEL` | Service log level (default `INFO`) |

## What gets recorded

Analytics Realtime writes several InfluxDB measurements:

| Measurement | Source endpoint |
|-------------|-----------------|
| `analytics` | Chat completions and embeddings |
| `rate_analytics` | The `/rate` endpoint |
| `mcp_analytics` | Tool set and application MCP endpoints |
| `routes_analytics` | Route endpoints |

The `analytics` measurement carries token and cost fields including `prompt_tokens`, `cached_prompt_tokens`, `completion_tokens`, `deployment_price` (this request only), and `price` (total including indirectly triggered requests). It also records the `trace_id`, `core_span_id`, and `core_parent_span_id` tags, which link each usage record back to its [trace](tracing).

## Step 3: Visualize in Grafana

The Analytics Realtime repository ships aggregated Grafana dashboards (6-hour and monthly trend views) under `dashboards/customized/influxdb/`. Follow the README in that directory to import them.

:::note
The aggregated dashboards require manually creating InfluxDB buckets and tasks — this is not automated by the Helm chart — and they support InfluxDB 2 only.
:::

## Result

Token usage, cost, and topic metrics for every successful request land in InfluxDB and render in the imported Grafana dashboards.
