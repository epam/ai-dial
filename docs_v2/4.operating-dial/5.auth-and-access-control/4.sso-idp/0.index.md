# SSO / IdP setup overview

DIAL authenticates end users through an external identity provider (IdP) that issues signed JWT access tokens. This page compares the supported providers and explains the shared setup workflow, so you can choose one without opening six guides. It is written for DevOps engineers who already understand [JWT-based authentication in DIAL](../jwt).

## How DIAL works with an IdP

DIAL does not store user passwords. Instead, DIAL Chat redirects users to your IdP to log in, the IdP returns a JWT, and DIAL Core validates that token and reads its claims on every request. Two components are involved:

- **DIAL Chat** initiates the OIDC login flow. It reads the provider's client ID, secret, and host from environment variables.
- **DIAL Core** validates the token signature against the provider's JWKS endpoint and maps a role claim to DIAL roles. It reads this from the `identityProviders` section of `settings.json`.

Every provider follows the same three-stage workflow, detailed in each guide:

1. **Configure the IdP** — register an application, capture its client ID, secret, and endpoints, and (optionally) create groups or roles.
2. **Configure DIAL** — set DIAL Chat environment variables and the DIAL Core `identityProviders.<name>` block in `settings.json`.
3. **Assign roles** — map the IdP's group or role claim values to DIAL resources through `userRoles` in `config.json`.

## Comparison of supported providers

| Provider | Protocol | Hosting | Role-claim source | DIAL Core `rolePath` |
|---|---|---|---|---|
| [AWS Cognito](aws-cognito) | OIDC | Managed (AWS) | Cognito groups | `cognito:groups` |
| [Auth0](auth0) | OIDC | Managed (SaaS) | Custom claim from a post-login action | `dial_roles` |
| [Google Identity](google-identity) | OIDC / OAuth 2.0 | Managed (Google) | Google Workspace groups (Cloud Identity API) | `fn:getGoogleWorkspaceGroups` |
| [Microsoft Entra ID](microsoft-entra-id) | OIDC | Managed (Microsoft) | Entra ID security groups | `groups` |
| [Okta](okta) | OIDC | Managed (SaaS) | Okta groups | `Groups` |
| [Keycloak](keycloak) | OIDC | Self-hosted (open source; bundled in the DIAL Helm chart) | Client roles via groups | `resource_access.<client>.roles` |
| [Azure AD B2C](azure-b2c) | OIDC | Managed (Microsoft CIAM) | Custom claim from a User Flow | `roles` |

## Choosing a provider

- **You already run one of these.** Use it. DIAL imposes no preference; the validation model is identical across providers.
- **You want a self-hosted, no-extra-cost option.** Keycloak ships inside the DIAL Helm chart and is the default for local and on-premises deployments.
- **You serve external or consumer users.** Azure AD B2C targets customer identity (CIAM) and supports custom sign-up flows.
- **You are standardized on a cloud.** AWS Cognito, Google Identity, and Microsoft Entra ID integrate naturally with their respective clouds' group management.

Group and role membership is configured entirely in the IdP. DIAL reads whatever claim you point `rolePath` at; it does not manage the directory.
