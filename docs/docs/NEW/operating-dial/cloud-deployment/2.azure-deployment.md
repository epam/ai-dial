# Deploy DIAL on Azure

Deploy the full DIAL stack to Azure Kubernetes Service (AKS) using the umbrella Helm chart, with Azure Blob Storage for files and Azure OpenAI as the model provider. This guide assumes you operate an AKS cluster and are familiar with Helm and `kubectl`. For the chart model and infrastructure mapping, read the [Helm chart overview](index) first.

## Prerequisites

- An AKS cluster, version 1.24 or higher, with `kubectl` and Helm 3.8.0 or higher configured against it
- The [Ingress-NGINX controller](https://kubernetes.github.io/ingress-nginx/) and [cert-manager](https://cert-manager.io/) installed in the cluster
- [Azure AD Workload Identity](https://azure.github.io/azure-workload-identity/) enabled on the cluster
- An Azure Blob Storage account for DIAL file storage
- An Azure OpenAI resource with at least one model deployed
- A domain and a cert-manager `ClusterIssuer` for TLS
- The DIAL Helm repository added:

  ```bash
  helm repo add dial https://charts.dialx.ai
  helm repo update
  ```

## Step 1: Create a values file

Start from the upstream [Azure `simple` example](https://github.com/epam/ai-dial-helm/tree/main/charts/dial/examples/azure/simple) and fill in your environment. The required values are:

```yaml
core:
  # Workload Identity managed identity client ID
  serviceAccount:
    annotations:
      azure.workload.identity/client-id: <AZURE_CORE_CLIENT_ID>
  env:
    aidial.storage.account: <AZURE_CORE_BLOB_STORAGE_NAME>
    aidial.storage.endpoint: <AZURE_CORE_BLOB_STORAGE_ENDPOINT>

openai:
  env:
    aidial.openai.endpoint: <AZURE_MODEL_ENDPOINT>
    aidial.openai.key: <AZURE_MODEL_KEY>

ingress:
  enabled: true
  hosts:
    - host: <DOMAIN>
  tls:
    - secretName: dial-tls
      hosts:
        - <DOMAIN>
  annotations:
    cert-manager.io/cluster-issuer: <CLUSTER_ISSUER>
```

Replace the placeholders:

- `<AZURE_CORE_CLIENT_ID>` — the Workload Identity managed identity client ID for DIAL Core
- `<AZURE_CORE_BLOB_STORAGE_NAME>` — the storage account name
- `<AZURE_CORE_BLOB_STORAGE_ENDPOINT>` — the Blob Storage endpoint URL
- `<AZURE_MODEL_ENDPOINT>` — the Azure OpenAI deployment endpoint
- `<AZURE_MODEL_KEY>` — the Azure OpenAI API key
- `<DOMAIN>` — your domain, for example `dial.example.com`
- `<CLUSTER_ISSUER>` — the cert-manager issuer, for example `letsencrypt-production`

Generate the DIAL API key, NextAuth secret, and encryption keys before installing. For production, read these from Azure Key Vault rather than inlining them — see [Azure Secrets deployment](azure-secrets-deployment).

## Step 2: Install the chart

```bash
helm install dial dial/dial \
  -f values.yaml \
  --version <CHART_VERSION> \
  --namespace dial \
  --create-namespace
```

Pin `<CHART_VERSION>` to a released chart version.

## Step 3: Verify the deployment

Check that the pods are running:

```bash
kubectl get pods -n dial
```

Once the ingress has an address and the certificate is issued, DIAL Chat is reachable at `https://<DOMAIN>/`. Confirm the Unified API responds:

```bash
curl https://<DOMAIN>/openai/deployments \
  -H "Api-Key: <YOUR_DIAL_API_KEY>"
```

## Result

DIAL Core, DIAL Chat, Redis, and the Azure OpenAI adapter run on your AKS cluster, with files persisted to Azure Blob Storage and models served from Azure OpenAI.

## Production architecture

The `simple` example is a starting point. For a production deployment, size the Azure services and configure redundancy as follows.

### Recommended service tiers

| Azure service | Tier |
|---------------|------|
| Azure Kubernetes Service | Standard, zone-redundant |
| Load Balancer | Standard |
| Storage Account | Standard, Azure Blob Storage, GZRS |
| Azure Cache for Redis | Premium P1, zone-redundant |
| Azure OpenAI | Sized to capacity requirements; provisioned throughput or pay-as-you-go |

Dependencies whose technology may follow corporate standards:

| Dependency | Azure service |
|------------|---------------|
| SSO | Microsoft Entra ID |
| Firewall | Azure Application Gateway with WAF |
| Logging and monitoring | Azure Application Insights and Log Analytics |
| Container registry | Azure Container Registry |
| Secret storage | Azure Key Vault |

### Scalability

DIAL services are stateless, so they run as multiple concurrent copies. As a baseline, run two copies of each service on different nodes for high availability. The services are not resource-intensive and handle thousands of users without further scaling.

To absorb load spikes, configure autoscaling for DIAL Core based on CPU usage (for example, scale out when usage exceeds 80% of the CPU limit for one minute), and enable cluster autoscaling so new nodes are added when existing nodes lack capacity.

Scaling language models is the more common need, because a single model instance is often capped by request rate limits. With Azure OpenAI there are four levels of scaling, applied in order:

1. Increase the number of model instances in the same region.
2. Request a regional quota extension — each region has a fixed tokens-per-minute (TPM) quota per model type.
3. Create model instances in other regions.
4. Use multiple subscriptions to access model endpoints.

### High availability

Achieve high availability through zone redundancy: run DIAL service copies in a multi-zone AKS cluster and use highly available configurations for the supporting services.

| Service | Availability SLA | Resilient to zone-wide outages |
|---------|------------------|--------------------------------|
| Azure Kubernetes Service | 99.95% with zone redundancy | Yes, if configured explicitly |
| Key Vault | 99.99% | Yes |
| Storage Account | 99.99% (read), 99.9% (write) | Yes, if configured explicitly |
| Azure Cache for Redis | 99.9% with zone redundancy | Yes, if configured explicitly |
| Azure OpenAI | No SLA | — |

Because Azure OpenAI has no SLA, define multiple model endpoints in different regions. If one endpoint fails, DIAL automatically tries the next in the list.

### Disaster recovery

DIAL has two parts with separate recovery policies: DIAL Core and the optional DIAL Analytics Realtime component.

DIAL Core consists of stateless services plus a single permanent store on an Azure Storage account (Redis is a cache and can be safely restarted). For a region-wide failure:

- **Storage account** — with GZRS redundancy it can fail over to the secondary region. After failover it runs as ZRS in one region; set the mode back to GZRS explicitly to restore cross-region failover capability for the next event.
- **All other services** — being stateless, redeploy the full set in a secondary region and re-route the DIAL domain name to the new entry point. To meet a tight recovery time objective, keep a passive DIAL Core deployment in the secondary region and activate it on failure.

When the primary region is restored, there is no need to fail back.
