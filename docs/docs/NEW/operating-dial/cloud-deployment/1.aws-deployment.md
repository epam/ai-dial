# Deploy DIAL on AWS

Deploy the full DIAL stack to Amazon EKS using the umbrella Helm chart, with Amazon S3 for storage and Amazon Bedrock as the model provider. This guide assumes you operate an EKS cluster and are familiar with Helm and `kubectl`. For the chart model and infrastructure mapping, read the [Helm chart overview](index) first.

## Prerequisites

- An EKS cluster, version 1.24 or higher, with `kubectl` and Helm 3.8.0 or higher configured against it
- The [AWS Load Balancer Controller](https://kubernetes-sigs.github.io/aws-load-balancer-controller/) installed in the cluster
- An S3 bucket for DIAL file storage
- Amazon Bedrock enabled in your target region, with access granted to at least one model
- IAM roles for service accounts (IRSA) for DIAL Core's S3 and Bedrock access
- An ACM certificate for your domain (for HTTPS ingress)
- The DIAL Helm repository added:

  ```bash
  helm repo add dial https://charts.dialx.ai
  helm repo update
  ```

## Step 1: Create a values file

Start from the upstream [AWS `simple` example](https://github.com/epam/ai-dial-helm/tree/main/charts/dial/examples/aws/simple) and fill in your environment. The required values are:

```yaml
core:
  # IRSA role granting access to the S3 bucket
  serviceAccount:
    annotations:
      eks.amazonaws.com/role-arn: <AWS_CORE_ROLE_ARN>
  env:
    aidial.storage.bucket: <AWS_CORE_S3_BUCKET_NAME>

bedrock:
  # IRSA role granting access to Amazon Bedrock
  serviceAccount:
    annotations:
      eks.amazonaws.com/role-arn: <AWS_BEDROCK_ROLE_ARN>
  env:
    aidial.bedrock.region: <AWS_BEDROCK_REGION>

ingress:
  enabled: true
  hosts:
    - host: <DOMAIN>
  tls:
    - secretName: dial-tls
      hosts:
        - <DOMAIN>
  annotations:
    alb.ingress.kubernetes.io/certificate-arn: <AWS_CERTIFICATE_ARN>
```

Replace the placeholders:

- `<AWS_CORE_ROLE_ARN>` — IRSA role ARN for S3 access
- `<AWS_CORE_S3_BUCKET_NAME>` — the S3 bucket name
- `<AWS_BEDROCK_ROLE_ARN>` — IRSA role ARN for Bedrock access
- `<AWS_BEDROCK_REGION>` — the region where Bedrock models are available
- `<DOMAIN>` — your domain, for example `dial.example.com`
- `<AWS_CERTIFICATE_ARN>` — the ACM certificate ARN for HTTPS

Generate the DIAL API key, encryption key, and Redis password before installing and supply them as values or [Kubernetes secrets](../configuration/core/settings-json/encryption).

## Step 2: Install the chart

```bash
helm install dial dial/dial \
  -f values.yaml \
  --version <CHART_VERSION> \
  --namespace dial \
  --create-namespace
```

Pin `<CHART_VERSION>` to a released chart version.

## Step 3: Verify the deployment

Check that the pods are running:

```bash
kubectl get pods -n dial
```

Once the load balancer has an address, DIAL Chat is reachable at `https://<DOMAIN>/` and the Unified API at the same host. Confirm the API responds:

```bash
curl https://<DOMAIN>/openai/deployments \
  -H "Api-Key: <YOUR_DIAL_API_KEY>"
```

## Result

DIAL Core, DIAL Chat, Redis, and the Bedrock adapter run on your EKS cluster, with files persisted to S3 and models served from Bedrock.

## Production architecture

The `simple` example is a starting point. For a production deployment, size the AWS services and configure redundancy as follows.

### Recommended service tiers

| AWS service | Configuration / tier |
|-------------|----------------------|
| Amazon EKS | Standard, zone-redundant |
| Elastic Load Balancing (ALB) | Standard |
| Amazon S3 | S3 Standard, with Multi-Region Access Points |
| ElastiCache | Standard, Multi-AZ |
| Amazon Bedrock | Sized to capacity requirements; usage-based pricing |

Dependencies whose technology may follow corporate standards:

| Dependency | AWS service |
|------------|-------------|
| SSO | Amazon Cognito |
| Firewall | AWS WAF with ALB or CloudFront, plus VPC security groups |
| Logging and monitoring | Amazon CloudWatch and AWS CloudTrail |
| Container registry | Amazon ECR |
| Secret storage | AWS Secrets Manager |

### Scalability

DIAL services are stateless, so they run as multiple concurrent copies. As a baseline, run two copies of each service on different nodes for high availability. The services are not resource-intensive and handle thousands of users without further scaling.

To absorb load spikes, configure autoscaling for DIAL Core based on CPU usage (for example, scale out when usage exceeds 80% of the CPU limit for one minute), and enable cluster autoscaling so new nodes are added when existing nodes lack capacity.

Scaling language models is the more common need, because a single Bedrock model instance is often capped by request rate limits. Options, in order of preference:

1. Request a regional quota increase through the AWS Service Quotas console or AWS Support.
2. Invoke models in additional regions, optionally with [cross-region inference](https://docs.aws.amazon.com/bedrock/latest/userguide/cross-region-inference.html).
3. Use multiple AWS accounts to access model endpoints.

### High availability

Achieve high availability through Multi-AZ deployments: run DIAL service copies across availability zones and use highly available configurations for the supporting services.

| Service | Availability SLA | Resilient to zone-wide outages |
|---------|------------------|--------------------------------|
| Amazon EKS | 99.95% | Yes, if configured explicitly |
| AWS IAM | 99.0%–99.99% | Yes |
| Amazon S3 | 99.9% | Yes, with multi-region replication |
| ElastiCache (Redis OSS) | 99.99% with Multi-AZ | Yes, if configured explicitly |
| Amazon Bedrock | 99.9% | Depends on configuration |
| AWS Secrets Manager | 99.99% | Yes |
| Amazon ECR | 99.9% | Yes (regional) |

Because Bedrock availability depends on configuration, define multiple model endpoints in different regions. If one region becomes unavailable, DIAL automatically tries the next endpoint in the list.

### Disaster recovery

DIAL has two parts with separate recovery policies: DIAL Core and the optional DIAL Analytics Realtime component.

DIAL Core consists of stateless services plus a single permanent store on S3 (ElastiCache is a cache and can be safely restarted). For a region-wide failure:

- **S3** — use Cross-Region Replication or Multi-Region Access Points for built-in cross-region redundancy. After a regional failure, review and re-establish the multi-region configuration as needed.
- **All other services** — being stateless, redeploy the full set in a secondary region and re-route the DIAL domain name to the new entry point. To meet a tight recovery time objective, keep a passive DIAL Core deployment in the secondary region and activate it on failure.

When the primary region is restored, there is no need to fail back.
