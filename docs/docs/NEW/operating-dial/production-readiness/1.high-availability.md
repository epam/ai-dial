# Run DIAL with high availability

Configure DIAL so a single node, pod, or availability-zone failure causes no outage. This guide covers the Helm chart settings that make the stateless services redundant, the managed cache choice for Redis, and model-endpoint failover. It assumes you deploy DIAL with the [umbrella Helm chart](/docs/NEW/operating-dial/cloud-deployment/aws-deployment) and have `kubectl` and Helm access.

The chart ships single-replica defaults suited to evaluation, not production. High availability is opt-in: you raise replica counts, spread pods across failure domains, and point Redis at a managed, replicated cache.

## Prerequisites

- DIAL deployed with the `dial` umbrella chart on a multi-node, multi-zone cluster
- A values file you can edit and apply with `helm upgrade`
- A managed Redis service (Amazon ElastiCache, Azure Cache for Redis, or Google Memorystore) for production

## Step 1: Run multiple replicas of each service

DIAL Core, DIAL Chat, and the adapters are stateless, so several copies share load and cover for each other. The chart default is `replicaCount: 1`. Set at least two for every service:

```yaml
core:
  replicaCount: 2
chat:
  replicaCount: 2
openai:
  replicaCount: 2
```

Rolling updates are the default deployment strategy (`updateStrategy.type: RollingUpdate`), so replica changes apply without downtime.

## Step 2: Spread pods across nodes and zones

Two replicas on the same node or zone do not survive that node or zone failing. Spread them with topology spread constraints or affinity rules:

```yaml
core:
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app.kubernetes.io/name: dial-core
```

The same keys (`topologySpreadConstraints`, `affinity`, `nodeSelector`, `tolerations`) exist on every service in the chart.

## Step 3: Keep a minimum available during disruptions

A Pod Disruption Budget stops voluntary disruptions — node drains, autoscaler scale-downs — from taking every replica at once. It is disabled by default (`pdb.create: false`). Enable it:

```yaml
core:
  pdb:
    create: true
    minAvailable: 1
```

## Step 4: Use a managed, replicated cache for Redis

The chart bundles a Bitnami `redis-cluster` subchart (`cluster.nodes: 3`) for convenience, but production deployments disable it and use a managed, highly available cache instead. Set `redis.enabled: false` and point DIAL Core at the managed endpoint:

```yaml
core:
  redis:
    enabled: false
  env:
    aidial.redis.provider.name: aws-elasti-cache   # or gcp-memory-store, azure-redis-cache
    aidial.redis.clusterServersConfig.nodeAddresses: '["redis://<MANAGED_ENDPOINT>:6379"]'
```

Choose the managed service's multi-AZ or zone-redundant tier. Because Redis is a cache, a failover or cold start costs a brief warm-up, not data.

## Step 5: Make model access highly available

A single model endpoint is a single point of failure and is often rate-limited. In `config.json`, give each model multiple `upstreams`. DIAL Core's load balancer spreads requests across them and fails over to the next endpoint when one is unavailable:

```json
{
  "models": {
    "gpt-4": {
      "type": "chat",
      "upstreams": [
        { "endpoint": "https://<REGION_A_ENDPOINT>/chat/completions" },
        { "endpoint": "https://<REGION_B_ENDPOINT>/chat/completions" }
      ]
    }
  }
}
```

## Step 6: Apply and verify

```bash
helm upgrade dial dial/dial -f values.yaml --version <CHART_VERSION> --namespace dial
kubectl get pods -n dial -o wide
```

Confirm replicas land on different nodes and zones in the `-o wide` output.

## Result

Each DIAL service runs redundant pods spread across failure domains, a Pod Disruption Budget preserves capacity during maintenance, the cache is a managed replicated service, and models fail over across endpoints. A single node or zone loss no longer causes an outage.
