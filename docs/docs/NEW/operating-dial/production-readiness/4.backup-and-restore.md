# Back up and restore DIAL

Protect DIAL's permanent data so you can recover from accidental deletion, corruption, or a region failure. This guide identifies exactly what to back up, how to use object-store features to do it, and how to restore. It assumes a Helm deployment using cloud object storage for the blob store, with access to your cloud console and `kubectl`.

The key insight from [the production-readiness model](/docs/NEW/operating-dial/production-readiness/index): DIAL services are stateless and the cache is disposable, so backup is narrowly about the permanent stores, not the whole cluster.

## Prerequisites

- DIAL blob storage on cloud object storage (Amazon S3, Azure Blob Storage, or Google Cloud Storage)
- Permission to enable versioning and replication on that bucket
- Access to your secret manager and, if you run Keycloak, to its Postgres database

## What to back up

| Target | Why | What happens if lost |
|--------|-----|----------------------|
| Blob storage bucket | Permanent data: conversations, files, published resources, app state | Permanent data loss |
| Encryption secret and key | Decrypts the blob data | Blob backups become unreadable |
| `config.json` and `settings.json` | Models, roles, keys, and Core settings | Manual reconstruction of configuration |
| Postgres database (only if Keycloak is used) | Realm, users, and client configuration | Identity provider rebuild |
| Redis | Cache only | Nothing — it rebuilds from blob storage |

:::warning
An encrypted blob backup is worthless without the encryption secret and key, and the key cannot be changed after first install. Back the key up to a secret manager, and verify a restore can actually read the data. See [Manage secrets](/docs/NEW/operating-dial/production-readiness/secrets-management).
:::

## Step 1: Protect the blob storage bucket

Use the object store's native durability features rather than copying data out:

- Enable **object versioning** so overwrites and deletes are recoverable.
- Enable **cross-region replication** so a regional failure does not lose data.
- Set lifecycle rules to expire old versions on a schedule that matches your retention policy.

These are configured in your cloud console, not in DIAL. The per-cloud guides describe the recommended setup — for example, [AWS disaster recovery](/docs/NEW/operating-dial/cloud-deployment/aws-deployment#disaster-recovery).

## Step 2: Back up configuration and the encryption key

Store `config.json`, `settings.json`, and the encryption secret and key in version control or a secret manager — never only in the live cluster. If you supply configuration through the Secrets Store CSI driver, the source of truth already lives in your secret manager; confirm it is itself backed up.

## Step 3: Back up Postgres (only if you run Keycloak)

DIAL Core itself needs no relational database. If you deploy the bundled Keycloak, its Postgres holds your realm. Dump it on a schedule:

```bash
kubectl exec -n dial <POSTGRES_POD> -- \
  pg_dumpall --clean --if-exists --quote-all-identifiers --no-password \
  > keycloak-$(date '+%Y-%m-%d').pgdump
```

## Step 4: Restore

To restore into a new or repaired environment:

1. Provision blob storage from the versioned or replicated copy, or restore the bucket from the target version.
2. Restore the encryption secret and key into the cluster or secret manager **before** starting DIAL Core.
3. Apply `config.json` and `settings.json`.
4. If you use Keycloak, restore Postgres from the dump.
5. Deploy DIAL with `helm upgrade`/`helm install`. The stateless services start clean and read permanent data from blob storage; Redis repopulates on demand.

:::warning
Helm does not delete PersistentVolumeClaims created by StatefulSets when you uninstall a release. Before reusing a namespace, check for and clean up orphaned PVCs so a restore does not bind to stale data.
:::

## Result

DIAL's permanent data is recoverable: the blob store is versioned and replicated, the encryption key and configuration are backed up off-cluster, and (if applicable) Keycloak's database has scheduled dumps. A restore brings the platform back without relying on the disposable cache.
