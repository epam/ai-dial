# Configure metrics and monitoring

This guide exposes DIAL metrics to Prometheus and wires them into a Prometheus Operator deployment on Kubernetes. It is for DevOps engineers running DIAL on Kubernetes with the [DIAL Helm chart](/docs/NEW/operating-dial/cloud-deployment/index). For the conceptual model, see [Observability](index).

## Prerequisites

- DIAL deployed with the `dial-core` Helm chart
- Prometheus running in the cluster, ideally via the [Prometheus Operator](https://prometheus-operator.dev/)
- Access to edit the chart's `values.yaml`

## The metrics endpoint

Each instrumented DIAL component serves Prometheus-format metrics on port `9464` at path `/metrics`. The port is exposed on the container but the metrics service is disabled until you enable it.

## Enable the metrics service

In the `dial-core` chart values, turn on metrics and the Prometheus Operator `ServiceMonitor`:

```yaml
metrics:
  enabled: true
  serviceMonitor:
    enabled: true        # creates a ServiceMonitor for the Prometheus Operator
    interval: 30s
    path: /metrics
    port: http-metrics
containerPorts:
  metrics: 9464          # change only if 9464 conflicts in your cluster
```

With `metrics.enabled: true`, the chart renders a dedicated `ClusterIP` service on the metrics port. With `serviceMonitor.enabled: true`, it renders a `ServiceMonitor` so the Prometheus Operator discovers and scrapes the endpoint automatically.

If you do not run the Prometheus Operator, leave `serviceMonitor.enabled: false` and add a static scrape job in your Prometheus configuration that targets the metrics service on port `9464`.

## Enable the exporter on the component

The endpoint only serves data when the component's OpenTelemetry instrumentation exports Prometheus metrics. Set the metrics exporter to include `prometheus`:

```bash
OTEL_METRICS_EXPORTER=otlp,prometheus
OTEL_EXPORTER_PROMETHEUS_PORT=9464
```

This exports metrics to your OTLP collector and serves the Prometheus scrape endpoint at the same time. Use `prometheus` alone if you do not run a collector.

## What gets scraped

:::warning
DIAL does not define custom Prometheus metric names. The scraped series are the standard metrics produced by the instrumentation layer: JVM and HTTP server metrics from the OpenTelemetry Java agent on DIAL Core, and system metrics (CPU, memory) plus FastAPI request metrics from the [DIAL SDK](/docs/NEW/building-with-dial/developer-tools/sdk-reference/telemetry) on Python adapters and applications. Do not build dashboards or alerts around DIAL-specific metric names — they do not exist.
:::

## Visualize in Grafana

:::note
The DIAL Helm chart ships no prebuilt Grafana dashboard for these Prometheus metrics. Use a community dashboard for the underlying instrumentation — for example a JVM or OpenTelemetry Java agent dashboard for DIAL Core — and adapt it to your labels. Prebuilt InfluxDB dashboards exist for the separate usage-analytics plane; see [Analytics Realtime setup](analytics-realtime-setup).
:::

## Result

Prometheus scrapes DIAL components on port `9464`, and the series appear in your Prometheus instance and any Grafana datasource that reads from it.
