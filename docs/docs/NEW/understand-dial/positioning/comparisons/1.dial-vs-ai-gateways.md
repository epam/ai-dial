# DIAL vs AI gateways

AI gateways such as LiteLLM, Portkey, and Kong AI Gateway sit between client applications and language model providers, normalizing many provider APIs behind one interface and adding routing, caching, rate limiting, and observability. DIAL overlaps with this layer but extends well beyond it. This page explains where DIAL and AI gateways do the same job, where they differ, and when to use each. It assumes familiarity with [What is DIAL](/docs/NEW/understand-dial/positioning/what-is-dial).

## What AI gateways do

An AI gateway is a proxy and control point for AI traffic. It gives a team one OpenAI-style API across many providers, plus operational features layered on top of that proxy:

- **LiteLLM** — open-source unified API and proxy server covering 100+ providers, with spend tracking. Lightweight and developer-friendly.
- **Portkey** — a production gateway with deep observability, 60+ guardrails, and prompt management, offered as a managed service over an open-source core.
- **Kong AI Gateway** — AI-specific plugins on Kong's API gateway: universal LLM API, semantic routing and caching, token- and cost-based load balancing, PII sanitization, and MCP traffic governance.

These tools do the gateway slice well and are often the right standalone choice when routing is the only problem to solve.

## Where DIAL overlaps

DIAL Core, together with [adapters](/docs/NEW/building-with-dial/adapters/index), performs the same gateway functions:

- One [Unified API](/docs/NEW/understand-dial/positioning/what-is-dial) across providers and clouds.
- Routing and load balancing across model upstreams, regions, and subscriptions.
- Per-role rate limits, cost control, and usage limits.
- OpenTelemetry-based observability across the request path.

If you only compared the gateway slice, DIAL and a dedicated gateway would look similar.

## Where DIAL differs

A gateway stops at the proxy boundary. DIAL continues into building and operating the applications that sit behind that boundary:

- **Application server.** DIAL hosts first-class applications — Custom Apps, Quick Apps, and Code Apps — as deployed services. Gateways route to your apps; DIAL runs them.
- **Resource storage with access control.** DIAL stores conversations, prompts, files, and applications with role-based access control. Gateways do not manage these resources.
- **Collaboration.** Sharing, organization-wide publications, and administrator review workflows are built in.
- **End-user UI.** DIAL Chat gives users a production interface — conversations, Marketplace, applications, files. Gateways are headless.

## Capability comparison

| Capability | AI gateways | DIAL |
|---|---|---|
| Unified API across providers | Yes | Yes |
| Routing, load balancing, caching | Yes | Yes |
| Rate limits and cost control | Yes | Yes |
| Guardrails / request middleware | Yes (varies) | Yes (interceptors) |
| Application runtime / app server | No | Yes |
| Resource storage with access control | No | Yes |
| Sharing, publications, review | No | Yes |
| End-user Chat UI | No | Yes |
| Open-source and self-hostable | Varies | Yes |

## When to use which

Choose a **standalone gateway** when your only requirement is normalizing and governing model traffic, and the applications, storage, and UI live elsewhere. Choose **DIAL** when you also need to build, store, govern, and serve the applications behind the gateway as one platform.

The two are not mutually exclusive. DIAL can sit alongside an existing gateway, or replace the gateway slice while adding everything above it. DIAL aims to complement the gateway layer, not to argue it away.
