# DIAL vs AI studios

Vendor AI studios such as Azure AI Foundry and Google AI Studio give teams a console to build, test, and run AI applications within a single vendor's cloud and model family. This is the category closest to DIAL in breadth. The decisive difference is portability: DIAL is vendor-agnostic, open-source, and self-hostable. This page explains the overlap, the differences, and when each fits. It assumes familiarity with [What is DIAL](/docs/NEW/understand-dial/positioning/what-is-dial).

:::note
Azure AI Studio was renamed Azure AI Foundry at Microsoft Ignite 2024, and rebranded again to Microsoft Foundry at Ignite 2025. This page uses the current names; "Azure AI Studio" refers to the same lineage.
:::

## What AI studios do

An AI studio is a managed environment anchored to its vendor's ecosystem:

- **Azure AI Foundry (Microsoft Foundry)** — a unified environment to build and manage AI applications, agents, models, datasets, evaluations, and search within Azure. Agents are first-class, with managed retrieval and enterprise governance through Microsoft Entra.
- **Google AI Studio** — a browser-based environment to try Gemini models, design and test prompts, generate code, and build prototypes through the Gemini API.

Both are strong when a team is committed to that vendor's cloud and models, and wants deep, native integration.

## Where DIAL overlaps

Both DIAL and the studios provide a place to build with models, and Azure AI Foundry adds agent runtime and governance — so conceptually the studios are the nearest neighbor to DIAL's scope. All three let you reach models, build applications, and apply some governance.

## Where DIAL differs

The difference is structural, not feature-by-feature:

- **Vendor- and cloud-agnostic.** DIAL puts every provider behind one Unified API and runs on any cloud or on-prem. Studios are anchored to their own cloud and model family — Azure with OpenAI and partners, Google with Gemini.
- **Open-source and self-hostable.** DIAL is licensed under Apache 2.0 and runs in your own environment. Studios are proprietary managed services.
- **Portability and lock-in.** Building on DIAL keeps applications portable across providers and clouds. Building in a studio ties applications to that vendor.
- **Extensibility.** DIAL is API-first, so every capability is an extension point. Studio extensibility is bounded by the vendor's platform.

## Capability comparison

| Capability | AI studios | DIAL |
|---|---|---|
| Build and run AI applications | Yes | Yes |
| Agent runtime and governance | Yes (varies) | Yes |
| Reach models across many vendors | No (vendor's own) | Yes |
| Multi-cloud / on-prem | No | Yes |
| Open-source and self-hostable | No | Yes |
| Deep single-vendor integration | Yes | Via adapters |
| Avoids vendor lock-in | No | Yes |

## When to use which — and together

Choose an **AI studio** when your organization is committed to one vendor's cloud and wants the deepest native integration with that vendor's models and tooling. Choose **DIAL** when portability, multi-cloud reach, open-source control, or self-hosting matter.

The two can coexist. DIAL can front a studio's models as one more provider through an adapter, giving you multi-cloud reach and a consistent governance and UI layer on top of vendor-native capabilities. DIAL complements the studios by removing the lock-in, not by reproducing every vendor-specific feature.
