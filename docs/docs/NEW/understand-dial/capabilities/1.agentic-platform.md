# Agentic platform

DIAL is an agentic platform: any model or application available in it can act as a reusable building block in a larger solution. This page explains what "agent" means in DIAL, how agents compose into multi-agent applications, and why the design favors reuse over rebuilding. It is written for architects and developers who have read [What is DIAL](/docs/NEW/understand-dial/positioning/what-is-dial) and want to understand the platform's composition model.

## What "agent" means in DIAL

In DIAL, an *agent* is any conversational entity a user or another application can interact with through the [Unified API](/docs/NEW/understand-dial/architecture/unified-api-overview) — a language model, an [adapter](/docs/NEW/building-with-dial/adapters/index)-backed provider model, or a full [application](/docs/NEW/building-with-dial/apps/index). Because every agent speaks the same OpenAI-compatible protocol, agents are interchangeable: a client that can call a model can call an application, and an application can call another application the same way it calls a model.

This uniformity is the foundation of the agentic platform. There is no separate "tool protocol" to learn for each building block — the Unified API is the contract that lets agents be combined.

## Agents as building blocks

The platform encourages composition rather than duplication. Instead of reimplementing a capability that already exists, you call the agent that provides it. Two kinds of building blocks are most common:

- **DIAL applications.** Applications built with the [DIAL SDK](/docs/NEW/building-with-dial/developer-tools/sdk-reference/index) expose chat completion or embedding endpoints that conform to the Unified API. An application can encapsulate retrieval, business logic, or a multi-step workflow and then be reused wherever an agent is expected.
- **Models through adapters.** Adapters translate provider-specific APIs into the Unified API, so commercial, open-source, self-hosted, and fine-tuned models all appear as agents. See [supported providers](/docs/NEW/building-with-dial/adapters/supported-providers) for the full range.

Because access control travels with every agent, composing an application out of other agents does not bypass permissions: a caller can only use the building blocks it is authorized to reach.

## Composing multi-agent applications

DIAL supports both simple conversational agents and applications that coordinate several agents to solve a task. The [agent builders](/docs/NEW/building-with-dial/apps/index) included in the platform make this composition accessible at different levels of effort:

- **[Quick Apps](/docs/NEW/building-with-dial/apps/quick-apps/index)** are no-code orchestrators. A model that supports tool use acts as the orchestrator, routing each request to the most appropriate agent, tool set, or model — without any custom code.
- **[Code Apps](/docs/NEW/building-with-dial/apps/code-apps/index)** and **custom applications** express orchestration logic in Python when a workflow needs more control than a no-code composer provides.

A multi-agent application can select and assign agents to tasks automatically. For example, an orchestrator might route a factual question to a retrieval agent, an image request to a text-to-image application, and a calculation to a code-execution tool — all behind a single conversational entry point.

## Discovery, reuse, and sharing

Agents only deliver leverage if developers and users can find and reuse them. DIAL Chat's [Marketplace](/docs/NEW/building-with-dial/developer-tools/chat-customization/marketplace) lists every agent a user is permitted to see, based on their roles and permissions. From there, agents can be used directly in a conversation or composed into new applications.

[Collaboration and sharing](/docs/NEW/understand-dial/capabilities/collaboration-and-sharing) extends this further: an agent built by one team can be shared with specific colleagues or published to the whole organization, becoming a building block others reuse. The result is a growing library of capabilities rather than repeated, isolated implementations.

## Why this design

We treat models and applications as one interchangeable category — agents — so that the cost of adding a capability falls over time. Every new application both solves its own problem and becomes a building block for the next solution. Standardizing on the Unified API rather than a bespoke plugin protocol keeps that composition predictable: the same authentication, access control, and observability apply no matter how deeply agents are nested.
