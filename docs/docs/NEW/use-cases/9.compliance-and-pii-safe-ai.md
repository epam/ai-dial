# Compliance and PII-safe AI

In a regulated organization, the question is not whether AI is useful but whether it can be used without sending personal data to a third party, losing the audit trail, or bypassing access rules. This page explains why compliance is the gate AI must pass in many enterprises, how DIAL enforces redaction, audit, and access boundaries at the gateway, and what it can and cannot guarantee. It is for risk and compliance owners and for the platform team that would implement the controls.

## Why this matters

The fastest way to stop an AI project is a data-protection review it cannot pass. Personal data flowing to an external model, prompts and responses that no one can later reconstruct, and a tool that quietly sidesteps document permissions are each enough to halt a rollout. These are not edge cases in finance, healthcare, or the public sector; they are the default concerns.

Bolting compliance onto each application individually does not work, because it only takes one application that forgot to redact, or one integration that skipped logging, to create the breach. Compliance has to be a property of the path every request takes, enforced in one place, not a promise each team makes separately.

## What you can do with DIAL

Because every request passes through DIAL Core, Core is where these controls live, and they apply uniformly no matter which application or integration sent the request.

- **Redaction and policy at the boundary.** [Interceptors](../building-with-dial/interceptors/0.index.md) sit in the request path and can transform traffic before it reaches a model — anonymizing personal data, filtering content, or enforcing policy. The [PII-redacting interceptor tutorial](../building-with-dial/interceptors/1.tutorial-pii-interceptor.md) is the worked example, and the [interceptor examples](../building-with-dial/interceptors/4.examples.md) catalog covers anonymization, content filtering, and more.
- **Access as a boundary.** [Authentication and access control](../understand-dial/security-and-governance/1.authentication-and-access-control.md) ensures a request — including a retrieval query — can only reach what its user is permitted to see.
- **An audit trail.** [Activity and rollback](../administering-dial/audit/activity-and-rollback.md) records what happened and supports reverting changes, so actions are reconstructable.

## How you'd build it

- **Insert the controls.** [Interceptors](../building-with-dial/interceptors/0.index.md) explains how redaction and policy attach to the request path, and [PII-redacting interceptor](../building-with-dial/interceptors/1.tutorial-pii-interceptor.md) builds one.
- **Set the boundaries.** [Authentication and access control](../understand-dial/security-and-governance/1.authentication-and-access-control.md) defines who reaches what.
- **Harden the deployment.** [Security hardening](../operating-dial/production-readiness/7.security-hardening.md) covers production-grade configuration, including handling of personal data.
- **Keep the record.** [Activity and rollback](../administering-dial/audit/activity-and-rollback.md) is the audit surface, and the [compliance and legal FAQ](../administering-dial/10.compliance-and-legal-faq.md) answers the common policy questions.

## Why DIAL over the alternatives

Per-application compliance scales badly and fails on the first oversight. Provider-side data controls help but stop at that provider's boundary and give you no single audit trail across vendors. DIAL enforces redaction, access, and logging at the gateway every request crosses, so coverage does not depend on each team remembering. What DIAL provides is the enforcement point and the tools; it does not, by itself, make an organization compliant. Regulatory compliance is a program — DIAL makes the technical controls consistent and auditable rather than scattered and optional.
