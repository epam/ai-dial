# Tutorial: custom adapter

In this tutorial, you will build a model adapter that translates requests from the DIAL Unified API to an external provider with a non-OpenAI response format. You will deploy the adapter alongside a mock provider service, register it in DIAL Core, and test it from DIAL Chat. Basic Python and Docker knowledge is required — read [What are adapters](index) first for conceptual background.

## Prerequisites

- Python 3.11 or later installed
- Docker and Docker Compose (V2) installed
- A terminal and a text editor

## What you will build

A custom adapter that sits between DIAL Core and a mock AI provider. The mock provider exposes a proprietary JSON format — not the OpenAI protocol — so the adapter must translate in both directions:

```
User → DIAL Chat → DIAL Core → Your Adapter → Mock Provider
                                     ↓ (translate format)
User ← DIAL Chat ← DIAL Core ← Your Adapter ← Mock Provider
```

## Project structure

When complete, your project directory will look like this:

```
my-custom-adapter/
├── adapter/
│   ├── app.py               # DIALApp entry point — registers the adapter
│   ├── adapter.py            # ChatCompletion subclass — translates requests and responses
│   └── requirements.txt      # Python dependencies (aidial-sdk, httpx)
├── mock-provider/
│   ├── server.py             # FastAPI mock with a non-OpenAI response format
│   └── requirements.txt      # Python dependencies (fastapi, uvicorn)
├── Dockerfile                # Container image for the adapter
├── Dockerfile.mock           # Container image for the mock provider
├── docker-compose.yml        # Runs DIAL Core + adapter + mock provider + DIAL Chat
└── config.json               # DIAL Core configuration — registers the adapter as a model
```

## Step 1: Create the project directory

```bash
mkdir -p my-custom-adapter/adapter
mkdir -p my-custom-adapter/mock-provider
cd my-custom-adapter
```

## Step 2: Build the mock provider

The mock provider simulates an external AI service with a proprietary API format. It accepts a JSON payload with `prompt` and `max_length` fields, and returns a `generated_text` response.

Create the file `mock-provider/server.py`:

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()


class GenerateRequest(BaseModel):
    prompt: str
    max_length: int = 512


class GenerateResponse(BaseModel):
    generated_text: str
    tokens_used: int


@app.post("/generate")
async def generate(req: GenerateRequest) -> GenerateResponse:
    response_text = f"Mock response to: {req.prompt[:200]}"
    return GenerateResponse(
        generated_text=response_text,
        tokens_used=len(response_text.split()),
    )
```

Create the file `mock-provider/requirements.txt`:

```text
fastapi==0.115.0
uvicorn==0.30.0
```

**Verify:** The mock provider defines a `/generate` endpoint that accepts `{"prompt": "...", "max_length": 512}` and returns `{"generated_text": "...", "tokens_used": 42}`. This format is intentionally different from the OpenAI protocol.

## Step 3: Implement the adapter

The adapter subclasses `ChatCompletion` from DIAL SDK — the same base class that Custom Apps use. The difference is that the adapter calls an external API instead of implementing business logic.

Create the file `adapter/adapter.py`:

```python
import httpx
from aidial_sdk.chat_completion import ChatCompletion, Request, Response

MOCK_PROVIDER_URL = "http://mock-provider:8080/generate"


class MockProviderAdapter(ChatCompletion):
    async def chat_completion(
        self, request: Request, response: Response
    ) -> None:
        last_message = request.messages[-1]
        user_text = last_message.text() or ""

        # Translate from Unified API format to the mock provider's format
        provider_payload = {
            "prompt": user_text,
            "max_length": 512,
        }

        async with httpx.AsyncClient(timeout=30.0) as client:
            provider_response = await client.post(
                MOCK_PROVIDER_URL, json=provider_payload
            )
            provider_response.raise_for_status()

        result = provider_response.json()

        # Translate from the mock provider's format back to Unified API format
        with response.create_single_choice() as choice:
            choice.append_content(result["generated_text"])
```

**Verify:** `MockProviderAdapter` extends `ChatCompletion` and overrides `chat_completion`. It translates the DIAL request into the mock format, calls the mock provider, and writes the result back through the DIAL response object.

## Step 4: Create the application entry point

Create the file `adapter/app.py`:

```python
import uvicorn

from aidial_sdk import DIALApp
from adapter import MockProviderAdapter

app = DIALApp()
app.add_chat_completion("my-adapter", MockProviderAdapter())
```

This registers the adapter under the name `my-adapter`. DIAL Core will route requests to this endpoint.

Create the file `adapter/requirements.txt`:

```text
aidial-sdk==0.36.0
httpx==0.27.0
uvicorn==0.30.0
```

**Verify:** The entry point creates a `DIALApp` and registers the adapter at the path `/openai/deployments/my-adapter/chat/completions`.

## Step 5: Create the Dockerfiles

Create `Dockerfile` for the adapter:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY adapter/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY adapter/adapter.py adapter/app.py ./

EXPOSE 5000

CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "5000"]
```

Create `Dockerfile.mock` for the mock provider:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY mock-provider/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY mock-provider/server.py .

EXPOSE 8080

CMD ["uvicorn", "server:app", "--host", "0.0.0.0", "--port", "8080"]
```

## Step 6: Create the Docker Compose configuration

Create `docker-compose.yml`:

```yaml
services:
  core:
    image: epam/ai-dial-core:0.23.0
    platform: linux/amd64
    environment:
      JAVA_OPTS: "-Xmx256m"
      aidial.config.files: '["/opt/config/config.json"]'
    ports:
      - "8080:8080"
    volumes:
      - ./config.json:/opt/config/config.json

  adapter:
    build:
      context: .
      dockerfile: Dockerfile
    depends_on:
      - core
      - mock-provider

  mock-provider:
    build:
      context: .
      dockerfile: Dockerfile.mock

  chat:
    image: epam/ai-dial-chat:0.22.0
    platform: linux/amd64
    environment:
      NEXT_PUBLIC_DIAL_API_HOST: "http://core:8080"
    ports:
      - "3000:3000"
    depends_on:
      - core
```

## Step 7: Configure DIAL Core

Create `config.json`. This registers the adapter as a model deployment so that DIAL Core routes requests to it:

```json
{
  "models": {
    "mock-model": {
      "type": "chat",
      "displayName": "Mock Provider Model",
      "endpoint": "http://adapter:5000/openai/deployments/my-adapter/chat/completions"
    }
  }
}
```

The `endpoint` field points to the adapter container. When a client requests the `mock-model` deployment, DIAL Core forwards the request to this endpoint. The adapter translates it and calls the mock provider.

For production adapters with multiple upstreams, authentication, and load balancing, see the [config.json models reference](/docs/NEW/operating-dial/configuration/core/config-json/models).

## Step 8: Build and run

```bash
docker compose up --build
```

**Verify:** All four services start without errors. Look for these log lines:

- `mock-provider` logs: `Uvicorn running on http://0.0.0.0:8080`
- `adapter` logs: `Uvicorn running on http://0.0.0.0:5000`
- `core` logs: `Server started`
- `chat` logs: `Ready on http://0.0.0.0:3000`

## Step 9: Test the adapter

Open DIAL Chat at `http://localhost:3000` and select **Mock Provider Model** from the model list.

Send a message:

```
Hello, can you summarize the weather today?
```

You should see a response from the mock provider:

```
Mock response to: Hello, can you summarize the weather today?
```

You can also test with `curl`:

```bash
curl -X POST http://localhost:8080/openai/deployments/mock-model/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [{"role": "user", "content": "Hello from curl"}]
  }'
```

The response follows the Unified API format with the mock provider's text in the `content` field.

## What you learned

- Adapters and Custom Apps use the same DIAL SDK — both subclass `ChatCompletion` and register with `DIALApp`
- An adapter translates between the Unified API format and the external provider's format
- DIAL Core routes model requests to the adapter based on the `endpoint` in `config.json`
- The adapter runs in its own container and is independent of the provider it translates for
