# Sharing API

Grant other users or deployments access to a resource without exposing it to the Public space, using the DIAL Core Sharing API. This guide is for developers building sharing workflows on top of the [Unified API](https://dialx.ai/dial_api#tag/Sharing). You should understand DIAL [access control](../../operating-dial/auth-and-access-control/0.index.md) and how resources are stored in private spaces.

In DIAL, a **resource** is a conversation, prompt, file, or application, owned by a user or a deployment (an application or Tool Set). When created, a resource lives in its owner's private folder. Sharing grants specific recipients access to it while keeping it out of the public file space. When a user shares a resource, it becomes available in DIAL Chat and the [Marketplace](../developer-tools/chat-customization/6.marketplace.md), where recipients can discover and use it.

The Sharing API has two perspectives: the resource **owner** and the resource **recipient**. It also includes a separate set of endpoints for **deployments** that need to share resources with each other during a request.

:::note
Users can authenticate to the Sharing API with either a [JWT](../../operating-dial/auth-and-access-control/2.jwt.md) or an [API key](../../operating-dial/auth-and-access-control/1.api-keys.md). Deployments must use [per-request keys](5.per-request-keys.md).
:::

## Endpoints for resource owners

| Endpoint | Description |
|---|---|
| [`create`](https://dialx.ai/dial_api#tag/Sharing/operation/shareResource) | Share a resource by creating an `invitationLink` that contains an `invitationId`. Grant **READ** or **WRITE** access. Add the **SHARE** permission to let recipients re-share the resource with **READ** access; use `maxAcceptedUsers` to cap how many users can accept the link (unlimited by default). SHARE can only be combined with another permission, and re-sharing is limited to READ. The system does not support listing all recipients of a shared resource. |
| [`copy`](https://dialx.ai/dial_api#tag/Sharing/operation/copySharedResources) | Add permissions for a new resource. For example, when a new attachment is added to an already-shared conversation, this ensures recipients can access the updated resource, including the new attachment. |
| [`list`](https://dialx.ai/dial_api#tag/Sharing/operation/getSharedResources) | Query all resources you have shared. Send `"with": "others"` in the request body. When a conversation with attachments is shared, the attachments appear as separate shared resources. |
| [`revoke`](https://dialx.ai/dial_api#tag/Sharing/operation/revokeSharedResources) | Stop sharing a resource. The content becomes inaccessible to all recipients, and any associated invitation link stops working. |
| [`invitations`](https://dialx.ai/dial_api#tag/Sharing/operation/getInvitations) | List all active invitations you have created. Invitations are created by the `create` endpoint. |
| [`invitations/{invitationId}`](https://dialx.ai/dial_api#tag/Sharing/operation/deleteInvitation) | Use the **DELETE** method to revoke a specific invitation. |

:::note
When an owner or recipient changes a shared resource, DIAL Core uses the HTTP `ETag` to version the resource and resolve conflicts. Use the [Subscribe API](4.notifications.md#subscribe-api) to be notified of changes made by other users or deployments.
:::

## Endpoints for resource recipients

| Endpoint | Description |
|---|---|
| [`discard`](https://dialx.ai/dial_api#tag/Sharing/operation/discardSharedResources) | Stop using a shared resource. The resource is no longer marked as shared for the owner. |
| [`list`](https://dialx.ai/dial_api#tag/Sharing/operation/getSharedResources) | Query all resources shared with you. Send `"with": "me"` in the request body. |
| [`invitations/{invitationId}`](https://dialx.ai/dial_api#tag/Sharing/operation/getInvitation) | Use the **GET** method to view invitation details and to **accept** the invitation. |
| [`create`](https://dialx.ai/dial_api#tag/Sharing/operation/shareResource) | Re-share a resource you received with the **SHARE** permission, creating a new `invitationLink`. You can grant only **READ** access when re-sharing. |

## Workflow examples

### Basic workflow

1. The owner shares a resource with `create` and gives the invitation link to the recipient.
2. The recipient extracts the `invitationId` from the link and calls `invitations/{invitationId}` to accept the resource.
3. The recipient lists shared resources and their details with `list`.

### Revoke workflow

1. The owner shares a resource with `create` and gives the invitation link to the recipient.
2. The recipient accepts the resource with `invitations/{invitationId}`.
3. The owner revokes access with `revoke`.
4. The recipient can no longer access the resource.

### Re-share workflow

1. The owner shares a resource with `create`, including the **SHARE** permission, and gives the invitation link to the recipient.
2. The recipient accepts the resource with `invitations/{invitationId}`.
3. The recipient re-shares the resource with `create` and gives a new invitation link to a second recipient.
4. The second recipient accepts the resource with `invitations/{invitationId}`.
5. The second recipient lists shared resources with `list`.

## Re-sharing rules

The **SHARE** permission lets a recipient re-share a resource. A re-shared resource is always granted with **READ** access only. DIAL Core validates each re-share request and returns `400` when a rule is violated:

- The original owner must have shared the resource with the **SHARE** permission.
- The invitation must be within its time-to-live, governed by the `invitation_ttl` DIAL Core dynamic setting.
- The re-share permission set must not include **WRITE** — only READ is allowed for re-sharing.
- The number of accepted invitations must not exceed the limit set by the `max_accepted_users` dynamic setting or by `maxAcceptedUsers` in the original share request.

For these limits, see the [configuration reference](../../operating-dial/configuration/0.index.md).

## Sharing in DIAL Chat

Chat users can share resources, accept invitations, and view resources shared by or with them, all from the UI. When a conversation that includes attachments is shared, the attachments are shared as well and can be opened through the chat file manager. For the end-user workflow, see the DIAL Chat user guide.

## Sharing between deployments

DIAL includes a separate API for cases where deployments — applications and Tool Sets — need to share their private resources with each other to complete a task. For example, a Mind Map application may need to give DIAL RAG access to its files.

### Authorization

Deployments must authenticate to this API with [per-request keys](5.per-request-keys.md) only. A request authorized with an API key or JWT is rejected: `403 Unauthorized - Operation is only permitted by per request API key`.

### Time-to-live

The owner controls the time-to-live of granted permissions within the scope of the current request. The recipient can access the shared resource only during the request timeline.

### Re-sharing between deployments

Granted permissions are not transitive — a recipient deployment cannot re-share them. For example, if Mind Map grants DIAL RAG read access to a file, DIAL RAG cannot pass that access to a third deployment; it must be granted explicitly through the permissions API. Permissions granted by the owner are revoked automatically once the recipient completes the current request.

The exception is [interceptors](../interceptors/0.index.md): when a resource is shared with a deployment that has interceptors, those interceptors gain access to the resource as well.

### Endpoints

| Endpoint | Description |
|---|---|
| [`per-request-permissions/grant`](https://dialx.ai/dial_api#tag/Sharing/operation/grantPerRequestPermissions) | Share a resource by path with another deployment, identified by its ID. Grant **READ** and **WRITE** access. The recipient implicitly accepts the invitation. |
| [`per-request-permissions/revoke`](https://dialx.ai/dial_api#tag/Sharing/operation/revokePerRequestPermissions) | Explicitly revoke access. Until access is revoked, the recipient can reuse the permission on identical subsequent calls. |
| [`per-request-permissions/list`](https://dialx.ai/dial_api#tag/Sharing/operation/getPerRequestPermissions) | List permissions granted by or to a specific deployment. |

### Example

In this scenario, a Mind Map application shares READ and WRITE access to a folder with DIAL RAG:

1. Mind Map calls `per-request-permissions/grant` to share its folder `files/<mind_map_bucket>/appdata/dial-rag` with the DIAL RAG deployment `applications/<user_bucket>/dial-rag`, granting READ and WRITE:

   ```json
   {
     "resources": [
       {
         "url": "files/<mind_map_bucket>/appdata/dial-rag",
         "permissions": ["READ", "WRITE"]
       }
     ],
     "receiver": "applications/<user_bucket>/dial-rag"
   }
   ```

2. DIAL Core records the grant against Mind Map's per-request key.
3. Mind Map calls DIAL RAG to perform an action that needs the shared folder.
4. DIAL Core issues a new per-request key to DIAL RAG and, finding that Mind Map shared resources with it, records the grant against DIAL RAG's key.
5. DIAL RAG writes to the shared folder through DIAL Core.
6. When the DIAL RAG call completes, DIAL Core invalidates its per-request key and the associated permissions.

## Result

You can share resources with users and deployments, manage invitations, revoke access, support controlled re-sharing, and grant transient access between deployments during a request.
