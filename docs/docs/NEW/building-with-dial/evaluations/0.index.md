# Evaluations overview

DIAL provides an open-source evaluation toolkit for measuring retrieval and generation quality in RAG applications. This page explains what the toolkit contains, what metrics it produces, and how evaluation fits into the development lifecycle. It is aimed at developers building RAG applications on DIAL who have a basic understanding of retrieval-augmented generation.

## Why evaluation matters

A RAG application can look correct in a demo and fail silently in production. The model generates fluent responses whether or not the retrieved context supports them. Without systematic measurement, there is no way to distinguish a faithful answer from a hallucinated one.

Evaluation answers a different question from operational analytics. Analytics Realtime tracks _how the system performs_—token usage, latency, cost per request. Evaluation tracks _how good the answers are_—whether the retrieved chunks contain the right information and whether the model's response is faithful to that information.

Both are necessary. Analytics catches performance regressions. Evaluation catches quality regressions. Together, they close the feedback loop that turns a prototype into a production system.

## The RAG evaluation toolkit

The DIAL evaluation toolkit consists of two open-source components, both licensed under Apache-2.0.

### DIAL Log Parser

[DIAL Log Parser](https://github.com/epam/ai-dial-log-parser) (`aidial-log-parser`) reads raw DIAL Core log files and transforms them into structured Parquet datasets. It extracts the question, answer, context, and token usage from each request-response pair and partitions the output by deployment name and date.

The parsed logs serve as input to the evaluation library. They can also be used independently for usage analysis, debugging, and audit.

### DIAL RAG Eval

[DIAL RAG Eval](https://github.com/epam/ai-dial-rag-eval) (`aidial-rag-eval`) is a Python library that computes retrieval and generation quality metrics. It takes two datasets—ground truth data (authored by a human) and collected answers (produced by the RAG application)—and returns a detailed metrics report.

Retrieval metrics are computed algorithmically by matching ground truth facts to retrieved context chunks. Generation metrics use an LLM judge to evaluate whether the model's response is entailed by the source material.

## What you can evaluate

The toolkit produces two categories of metrics.

### Retrieval metrics

Retrieval metrics measure how well the retrieval step finds the right information. They do not require an LLM—computation is purely algorithmic.

| Metric | What it measures |
|---|---|
| Precision | Fraction of retrieved chunks that contain at least one ground truth fact |
| Recall | Fraction of ground truth facts found in any retrieved chunk |
| MRR (Mean Reciprocal Rank) | Average inverse rank of the first relevant chunk per fact—higher means relevant chunks appear earlier |
| F1 | Harmonic mean of precision and recall |

### Generation metrics

Generation metrics measure answer quality by evaluating entailment between text pairs. They require an LLM judge (the library uses any `langchain-core` compatible chat model).

| Metric | Premise → Hypothesis | What it measures |
|---|---|---|
| Context → Answer | Retrieved context → Generated answer | **Faithfulness**: can the answer be derived entirely from the context? |
| Answer → Ground Truth | Generated answer → Expected answer | **Coverage**: does the answer entail the expected answer? |
| Ground Truth → Answer | Expected answer → Generated answer | **Correctness**: does the expected answer entail the generated answer? |
| Answer → Facts | Generated answer → Ground truth facts | **Fact coverage**: does the answer cover all expected facts? |
| Facts → Answer | Ground truth facts → Generated answer | **Factual grounding**: are the answer's claims supported by the expected facts? |

The library also detects **answer refusal**—cases where the model explicitly declines to answer or states that information is unavailable.

Each inference metric uses a three-stage LLM pipeline: decontextualize sentences, extract atomic statements, then score each statement as entailed, contradicted, or neutral. The final score is the weighted average of entailed statements across all segments.

## How evaluation fits the development lifecycle

Evaluation is most effective as a recurring step in the development cycle, not a one-time check. The pattern—sometimes called _eval-driven development_—works as follows:

1. **Build** a RAG application and deploy it to a DIAL instance.
2. **Collect** responses by running a set of test questions through the application.
3. **Prepare** ground truth data—expected answers and the key facts each answer must contain.
4. **Evaluate** by running DIAL RAG Eval against the ground truth and collected answers.
5. **Analyze** the metrics to identify weaknesses (low recall means missing documents; low faithfulness means the model is hallucinating beyond the context).
6. **Improve** the application—adjust chunking strategy, change the embedding model, tune the top-k parameter, or refine the system prompt.
7. **Re-evaluate** to verify the improvement and catch regressions.

Each iteration tightens the quality of the RAG pipeline. Over time, the ground truth dataset grows into a regression test suite for the application.

## Quality evaluation vs operational analytics

| Aspect | Quality evaluation | Operational analytics |
|---|---|---|
| Question answered | Are the answers correct and faithful? | Is the system fast, reliable, and cost-effective? |
| Data source | Ground truth datasets (human-authored) | Production request logs |
| Timing | Offline, during development | Real-time, in production |
| Tool | DIAL RAG Eval | Analytics Realtime |
| Output | Precision, recall, faithfulness scores | Token usage, latency, cost per request |

Use quality evaluation during development and before releases. Use operational analytics in production to monitor ongoing performance.
