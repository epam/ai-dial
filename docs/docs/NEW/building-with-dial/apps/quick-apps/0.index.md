# What are Quick Apps

Quick Apps are a no-code orchestrator for building AI workflows on DIAL. They combine a language model, system instructions, and tools into a complete application — without writing code. Quick Apps are conceptually similar to OpenAI's GPTs.

:::tip Use Quick App 2.0 for new applications
DIAL provides two Quick App types. **Quick App 2.0** is the current recommended version with a richer configuration model and a visual wizard. The original **Quick App** is also fully supported. Each type has its own dedicated documentation section.
:::

This page explains how Quick Apps work, what tools they can use, and how they fit into the DIAL app ecosystem.

## How Quick Apps work

A Quick App is defined by three elements:

- **Model** — the language model that processes user messages and decides when to use tools.
- **Instructions** — a system prompt that guides the model's behavior, tone, and scope.
- **Tools** — external capabilities the model can invoke: other DIAL apps, web APIs, MCP servers, and more.

When a user sends a message, the model receives the system instructions and the conversation history. If the model determines that a tool is needed to answer the question, it calls the appropriate tool, receives the result, and incorporates it into the response. This loop can repeat multiple times within a single request.

Because Quick Apps follow the Unified API, they are indistinguishable from other DIAL apps to callers. A Quick App can be called by other apps, used as a tool in another Quick App, or accessed through DIAL Chat and the API.

## Available tools

Quick Apps support several types of tools, each connecting to a different kind of capability:

| Tool type | What it does | Example |
|---|---|---|
| **Apps as tools** | Calls other DIAL applications by ID or by matching conditions (regex on ID, display name, or description) | A Quick App that routes complex questions to a specialized RAG app |
| **Models as tools** | Calls other language models deployed in DIAL | A Quick App that uses one model for analysis and another for summarization |
| **Web API Tool Sets** | Calls external REST APIs with configurable authentication (API key, basic, bearer, OAuth 2.0) | A weather assistant that calls the Open-Meteo API |
| **MCP Tool Sets** | Connects to [MCP servers](./tool-sets/index) via SSE or HTTP streaming transport | A Quick App that accesses a database through an MCP server |
| **Client Tool Sets** | Defines tools that execute on the client side (browser), with state restoration metadata | A Quick App that triggers a client-side geocoding lookup |
| **RAG (document grounding)** | Grounds the model's responses in uploaded documents or file URLs | A Q&A assistant that answers questions from a specific PDF |

Tools can be combined freely. A single Quick App can use multiple tool types simultaneously.

## Creating Quick Apps

Quick Apps can be created through three methods:

| Method | Best for | Guide |
|---|---|---|
| **UI wizard in DIAL Chat** | End users and rapid prototyping. Visual form for model, instructions, and tools — no config files needed. | [Quick App 2.0](./quick-app-2/create-via-ui) · [Quick App](./quick-app-original/create-and-configure#create-via-ui) |
| **API** | Developers building automation, CI/CD pipelines, or programmatic app management. | [Quick App 2.0](./quick-app-2/create-via-api) · [Quick App](./quick-app-original/create-and-configure#create-via-api) |
| **config.json** | DevOps teams provisioning apps as infrastructure-as-code. Apps are available immediately when DIAL Core starts. | [Quick App 2.0](./quick-app-2/create-via-config) · [Quick App](./quick-app-original/create-and-configure#create-via-configjson) |

## Quick App 2.0 vs Quick App

| | Quick App 2.0 | Quick App (original) |
|---|---|---|
| **Schema ID** | `quickapps2` | `quickapps` |
| **Architecture** | Orchestrator-based (model + prompt + iteration limit) | Flat (`model`, `instructions`, `temperature` at top level) |
| **Tool configuration** | Single typed `tool_sets` array (6 types) | Separate arrays (`web_api_toolset`, `mcp_toolset`, `applications_as_tools`, `client_toolset`) |
| **System prompt** | 3 types (custom, DIAL file, predefined) with `{{variable}}` support | Plain string |
| **Agent loop control** | Explicit `max_iterations` (default 15) | Implicit |
| **Client-side tools** | Not supported | Supported |
| **Condition-based tool selection** | Not supported | Supported (regex on ID, name, description) |
| **Visual editor** | Full wizard in DIAL Chat | Limited form |
| **Executor service** | [ai-dial-quickapps-backend](https://github.com/epam/ai-dial-quickapps-backend) | Separate executor service |
| **Recommendation** | **Use for new apps** | Use when you need client tools or condition matching |

To migrate an existing Quick App to 2.0, see [Migrate to Quick App 2.0](./quick-app-original/migrate-to-2).

## Quick Apps 2.0

Quick Apps 2.0 is the current recommended version. It introduces an orchestrator-based architecture with a richer configuration model:

- **Orchestrator** — a dedicated deployment (model + system prompt) that drives the agent loop, decides when to invoke tools, and composes the final response.
- **Tool sets** — a typed array that supports six different connection types: REST APIs, DIAL deployments, external MCP servers, DIAL-managed MCP servers, internal tools, and predefined templates.
- **Contexts** — attach documents from DIAL storage as grounding sources for RAG, or inject structured content directly into the model's context.
- **Component-based architecture** — build workflows from reusable agent components that can be shared, published, and referenced across apps.
- **Visual editor** — compose workflows in a dedicated UI editor in DIAL Chat.

Quick Apps 2.0 is defined by the application type schema ID `https://mydial.epam.com/custom_application_schemas/quickapps2`. The three required top-level properties are:

| Property | Required | Description |
|---|---|---|
| `orchestrator` | Yes | Model, system prompt, and iteration limit for the orchestrator. |
| `contexts` | Yes | Array of document contexts for RAG grounding (may be empty). |
| `tool_sets` | Yes | Array of tool set configurations (may be empty). |
| `starters` | No | Suggested starting prompts shown to users. |

Minimal valid Quick App 2.0 configuration:

```json
{
  "orchestrator": {
    "deployment": {
      "name": "gpt-4o-mini",
      "parameters": { "temperature": 0.5 }
    },
    "system_prompt": {
      "type": "custom",
      "variables": {},
      "content": "You are a helpful assistant."
    }
  },
  "contexts": [],
  "tool_sets": []
}
```

## Quick App

The original Quick App uses a flat `applicationProperties` schema with fields like `model`, `instructions`, `temperature`, `starters`, and tool arrays.

Key properties of the legacy schema ID `https://mydial.epam.com/custom_application_schemas/quickapps`:

| Property | Required | Description |
|---|---|---|
| `temperature` | Yes | Controls response randomness (0.0–1.0) |
| `instructions` | Yes | System prompt that guides model behavior |
| `model` | Yes | Language model ID |
| `starters` | No | Suggested starting prompts shown to users |
| `applications_as_tools` | No | DIAL apps and models available as tools |
| `client_toolset` | No | Client-side tools with state restoration |
| `web_api_toolset` | No | External REST API integrations |
| `mcp_toolset` | No | MCP server connections |
| `document_relative_url` | No | File URL for RAG grounding |

See [Quick App configuration reference](./quick-app-original/reference) for the full schema documentation.

## Sharing and access

Quick Apps follow the standard DIAL sharing model:

- **Private by default** — a new Quick App is visible only to its creator.
- **Shareable** — grant READ or WRITE access to specific users. WRITE access enables real-time co-editing.
- **Publishable** — submit the app to the Marketplace for broader access. Publishing requires administrator approval.

## When Quick Apps are the right choice

Quick Apps work well when:

- You want to compose a workflow from existing models, apps, and APIs without writing code.
- You need rapid prototyping — create, test, and iterate in minutes.
- You want RAG grounding with minimal setup.
- You need tool orchestration where the model decides when and how to use tools.
- End users should be able to create their own customized assistants.

For use cases that require full network access, custom dependencies, or your own infrastructure, consider a [Custom App](../custom-apps/index) instead. For quick Python prototyping without infrastructure concerns, consider a [Code App](../code-apps/index).
